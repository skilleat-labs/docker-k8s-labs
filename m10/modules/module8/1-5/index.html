    1-5. Dockerfile 최적화 실습¶

    실습 목표¶

    • Dockerfile 작성 시 효율성과 보안을 고려한 최적화 기법 체험하기
    • RUN 명령어 최소화, 캐시 전략, .dockerignore 활용하기
    • 멀티 스테이지 빌드와 비-root 사용자 적용 실습

    예상 소요 시간 (약 30분)¶

    • 0~5분: 기본 Dockerfile 작성 및 빌드
    • 5~15분: RUN 최소화, 캐시 전략, .dockerignore 실습
    • 15~23분: 멀티 스테이지 빌드 실습
    • 23~30분: 비-root 보안 적용 및 정리

    사전 조건¶

    • Docker 설치 완료
    • 기본 터미널 명령어 이해

    확인:

    docker --version
docker ps

    1) 기본 Dockerfile 작성 및 빌드 체험¶

    1-1. 작업 디렉토리 준비¶

    mkdir -p ~/docker-optimize
cd ~/docker-optimize

    1-2. 간단한 Node.js 앱 준비¶

    파일 설명

    • app.js : 실제 서버 코드. "포트 3000에서 HTTP 요청을 받아 응답을 돌려준다"는 내용이 담긴 애플리케이션 파일입니다.
    • package.json : 이 앱이 어떤 외부 라이브러리(여기서는 express)에 의존하는지 명시하는 Node.js 프로젝트 설정 파일입니다. npm install 시 이 파일을 읽어 라이브러리를 설치합니다.

    app.js 파일 생성:

    cat > app.js <<'EOF'
const express = require('express');
const app = express();

app.get('/', (req, res) => {
  res.send('Hello from optimized Docker!');
});

app.listen(3000, () => {
  console.log('App running on port 3000');
});
EOF

    package.json 파일 생성:

    cat > package.json <<'EOF'
{
  "name": "docker-optimize",
  "version": "1.0.0",
  "main": "app.js",
  "scripts": {
    "start": "node app.js"
  },
  "dependencies": {
    "express": "^4.18.0"
  }
}
EOF

    1-3. 기본 Dockerfile 작성¶

    Dockerfile 생성:

    nano Dockerfile

    아래 내용을 직접 입력하세요:

    FROM node:18-alpine

WORKDIR /app

COPY package*.json ./
RUN npm install

COPY . .

EXPOSE 3000
CMD ["npm", "start"]

    저장: Ctrl + O → Enter → 종료: Ctrl + X

    1-4. 빌드 및 실행¶

    docker build -t myapp:v1 .
docker run -d --name myapp-v1 -p 3000:3000 myapp:v1
curl http://localhost:3000

    앱이 정상 동작하면 Hello from optimized Docker!가 출력됩니다.

    docker images | grep myapp

    이 크기와 빌드 시간을 이후 최적화 버전과 비교합니다.

    2) RUN 명령어 최소화 실습¶

    2-1. 비효율적인 Dockerfile 예시¶

    Dockerfile.bad 생성 (비교용):

    nano Dockerfile.bad

    아래 내용을 직접 입력하세요:

    FROM node:18-alpine

WORKDIR /app

RUN apk add --no-cache git
RUN apk add --no-cache curl
RUN apk add --no-cache vim

COPY package*.json ./
RUN npm install

RUN mkdir -p /app/logs
RUN mkdir -p /app/cache

COPY . .

EXPOSE 3000
CMD ["npm", "start"]

    저장: Ctrl + O → Enter → 종료: Ctrl + X

    2-2. 최적화된 Dockerfile 작성¶

    Dockerfile 수정 (레이어 통합):

    nano Dockerfile

    아래 내용으로 교체하세요:

    FROM node:18-alpine

WORKDIR /app

# RUN 명령어 통합 (레이어 수 최소화)
RUN apk add --no-cache git curl vim && \
    mkdir -p /app/logs /app/cache

COPY package*.json ./
RUN npm install

COPY . .

EXPOSE 3000
CMD ["npm", "start"]

    저장: Ctrl + O → Enter → 종료: Ctrl + X

    2-3. 빌드 비교¶

    # 비효율 Dockerfile 빌드
docker build -f Dockerfile.bad -t myapp:bad .

# 최적화 Dockerfile 빌드
docker build -t myapp:v2 .

# 크기 비교
docker images | grep myapp

    v1이 더 작은 이유

    docker images에 v1도 함께 보이는데, v1은 git/curl/vim을 설치하지 않았기 때문에 더 작습니다. 이 섹션의 비교 대상은 bad vs v2입니다. 두 이미지는 동일한 패키지를 설치하므로 크기가 같고, RUN 통합의 이점은 크기 감소가 아니라 레이어 수 감소입니다.

    레이어 수 확인:

    # bad: RUN 명령어가 6개 → 레이어 6개
docker history myapp:bad | head -15

# v2: RUN 명령어가 2개로 통합 → 레이어 2개
docker history myapp:v2 | head -15

    관찰 포인트:

    • myapp:bad는 레이어가 더 많음 → 각 레이어마다 메타데이터·캐시 파일 발생
    • myapp:v2는 레이어 수가 적어 이미지 배포 및 관리가 효율적
    • 만약 RUN 안에서 임시 파일을 삭제한다면(rm -rf /tmp/*) 그 때 실제 크기도 줄일 수 있음

    3) 캐시 전략 실습¶

    3-1. 캐시가 깨지는 비효율적인 순서¶

    Dockerfile.cache-bad 생성:

    nano Dockerfile.cache-bad

    아래 내용을 직접 입력하세요:

    FROM node:18-alpine

WORKDIR /app

# 소스 코드 먼저 복사 (캐시 비효율)
COPY . .

RUN npm install

EXPOSE 3000
CMD ["npm", "start"]

    저장: Ctrl + O → Enter → 종료: Ctrl + X

    3-2. 최적화된 캐시 순서¶

    Dockerfile 수정 (의존성 → 소스 코드 순서):

    nano Dockerfile

    아래 내용으로 교체하세요:

    FROM node:18-alpine

WORKDIR /app

# 의존성 파일 먼저 복사 (캐시 최적화)
COPY package*.json ./
RUN npm install

# 소스 코드 나중에 복사
COPY . .

EXPOSE 3000
CMD ["npm", "start"]

    저장: Ctrl + O → Enter → 종료: Ctrl + X

    3-3. 캐시 효과 체험¶

    먼저 소스 코드를 수정합니다 (실제 개발에서 코드가 바뀐 상황을 흉내냅니다):

    echo "// change" >> app.js

    비효율 순서(cache-bad)로 빌드:

    time docker build -f Dockerfile.cache-bad -t myapp:cache-bad .

    관찰 — npm install이 새로 실행되는지 확인

    빌드 로그에서 아래와 같이 CACHED가 없는 npm install 줄을 찾으세요.

    => RUN npm install          ← CACHED 없음 → 새로 실행 (30~60초)

    소스 코드가 바뀌었을 뿐인데 npm install이 매번 다시 실행됩니다.

    최적화 순서로 빌드:

    time docker build -t myapp:v4 .

    관찰 — npm install이 캐시에서 오는지 확인

    빌드 로그에서 아래 두 줄을 찾으세요.

    => CACHED RUN npm install   ← 캐시 재사용 → 즉시 통과
=>        COPY . .          ← 소스 변경분만 새로 복사

    npm install은 건드리지 않고, 변경된 소스 코드만 다시 복사합니다. 빌드 시간이 1초 미만으로 완료되는 것을 확인하세요.

    4) .dockerignore 활용 실습¶

    4-1. .dockerignore 파일 생성¶

    .dockerignore 생성:

    nano .dockerignore

    아래 내용을 직접 입력하세요:

    # Node.js
node_modules
npm-debug.log*

# Git
.git
.gitignore

# IDE
.vscode
.idea

# OS
.DS_Store
Thumbs.db

# Logs
*.log
logs/

# Runtime
.env
.env.local

# Build artifacts
dist/
build/

    저장: Ctrl + O → Enter → 종료: Ctrl + X

    4-2. 빌드 컨텍스트 크기 비교¶

    먼저 .dockerignore 없이 빌드합니다:

    mv .dockerignore .dockerignore.bak
docker build --no-cache -t myapp:v5 .

    관찰 — 빌드 로그에서 이 줄을 찾으세요

    => => transferring context: 390B    ← .dockerignore 없음
    이 숫자가 Docker 데몬에 전달되는 파일 묶음(빌드 컨텍스트)의 크기입니다.

    .dockerignore 적용 후 빌드합니다:

    mv .dockerignore.bak .dockerignore
docker build --no-cache -t myapp:v6 .

    관찰 — 컨텍스트 크기가 줄어드는지 확인

    => => transferring context: 196B    ← .dockerignore 적용
    .dockerignore에 명시한 파일들이 빌드 컨텍스트에서 제외되어 크기가 줄어듭니다.

    지금 프로젝트가 작아서 차이가 미미하지만, 실제 프로젝트에서 node_modules(수백 MB)를 제외하면 빌드 시작 속도가 극적으로 빨라집니다.

    5) 멀티 스테이지 빌드 실습¶

    5-1. 빌드와 실행 환경 분리¶

    Dockerfile.multi 생성:

    nano Dockerfile.multi

    아래 내용을 직접 입력하세요:

    # 빌드 스테이지
FROM node:18-alpine AS builder

WORKDIR /app

COPY package*.json ./
RUN npm ci --only=production

# 실행 스테이지 (경량화)
FROM node:18-alpine

RUN addgroup -g 1001 -S nodejs && \
    adduser -S nextjs -u 1001

WORKDIR /app

# 빌드 스테이지에서 필요한 파일만 복사
COPY --from=builder /app/node_modules ./node_modules
COPY . .

USER nextjs

EXPOSE 3000
CMD ["npm", "start"]

    저장: Ctrl + O → Enter → 종료: Ctrl + X

    5-2. 멀티 스테이지 빌드 실행¶

    docker build -f Dockerfile.multi -t myapp:multi .
docker images | grep myapp

    관찰 포인트:

    • 이미지 크기 대폭 감소 (빌드 도구 제외)
    • COPY --from=builder 구문 이해

    6) 비-root 보안 적용 실습¶

    6-1. root 위험성 체험¶

    기존 컨테이너에서 root 권한 확인:

    docker run --rm myapp:v1 whoami  # root로 실행됨

    6-2. 비-root 사용자 적용¶

    Dockerfile.secure 생성:

    nano Dockerfile.secure

    아래 내용을 직접 입력하세요:

    FROM node:18-alpine

# 비-root 사용자 생성
RUN addgroup -g 1001 -S appgroup && \
    adduser -S appuser -u 1001 -G appgroup

WORKDIR /app

# 권한 설정
RUN chown -R appuser:appgroup /app

# 의존성 설치
COPY package*.json ./
RUN npm install

# 소스 복사
COPY . .

# 사용자 전환
USER appuser

EXPOSE 3000
CMD ["npm", "start"]

    저장: Ctrl + O → Enter → 종료: Ctrl + X

    6-3. 보안 적용 빌드 및 실행¶

    docker build -f Dockerfile.secure -t myapp:secure .
docker run --rm myapp:secure whoami  # appuser로 실행됨
docker run -d --name myapp-secure -p 3001:3000 myapp:secure
curl http://localhost:3001

    관찰 포인트:

    • 최소 권한 원칙 적용
    • K8s 등 컨테이너 오케스트레이션 호환성 향상

    7) 정리 및 비교¶

    7-1. 모든 이미지 크기 비교¶

    docker images | grep myapp

    7-2. 컨테이너 정리¶

    docker rm -f $(docker ps -aq) 2>/dev/null || true
docker rmi myapp:v1 myapp:v2 myapp:v3 myapp:v4 myapp:v5 myapp:v6 myapp:multi myapp:secure 2>/dev/null || true

    7-3. 최적화 효과 요약¶

    버전 특징 이미지 크기 빌드 시간
    v1 기본 보통 보통
    v2 RUN 최소화 감소 비슷
    v3 캐시 최적화 비슷 단축
    multi 멀티 스테이지 대폭 감소 비슷
    secure 비-root 비슷 비슷
