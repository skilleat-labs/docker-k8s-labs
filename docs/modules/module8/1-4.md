## 1) 컨테이너 데이터 유실 문제 체험

### 1-1. 컨테이너 내 파일 생성 및 확인

```bash
docker run -it --name temp-test skilleatlab.azurecr.io/lab/busybox:latest
```

컨테이너 내부에서:

```bash
# 파일 생성
echo "This is important data" > /data.txt
cat /data.txt

# 디렉토리도 생성
mkdir -p /myapp/config
echo "config value=123" > /myapp/config/settings.txt

# 현재 상태 확인
ls -la /
ls -la /myapp/config

exit
```

### 1-2. 컨테이너 삭제 후 데이터 확인

```bash
# 컨테이너 삭제
docker rm temp-test

# 같은 이름으로 새 컨테이너 생성
docker run -it --name temp-test skilleatlab.azurecr.io/lab/busybox:latest
```

컨테이너 내부에서:

```bash
# 이전에 만든 파일이 있는지 확인
cat /data.txt  # 오류 발생
ls -la /myapp  # 디렉토리가 없음

exit
```

**관찰 포인트:**

- 새 컨테이너는 이미지의 깨끗한 상태로 시작
- 이전 컨테이너의 모든 변경사항이 사라짐
- 이것이 바로 "Ephemeral" 특성

```bash
docker rm temp-test
```

## 2) Docker Volume으로 데이터 보존하기

### 2-1. 볼륨 생성 및 확인

```bash
# 볼륨 생성
docker volume create mydata

# 생성된 볼륨 확인
docker volume ls

# 상세 정보 확인
docker volume inspect mydata
```

관찰 사항:

- `Mountpoint`: 호스트에서 실제 저장되는 위치
- 도커가 자동으로 관리하는 저장소

### 2-2. 볼륨을 마운트한 컨테이너 실행

```bash
docker run -it --name vol-test -v mydata:/data skilleatlab.azurecr.io/lab/busybox:latest
```

컨테이너 내부에서:

```bash
# /data 디렉토리에 여러 파일 생성
echo "Volume data 1" > /data/file1.txt
echo "Volume data 2" > /data/file2.txt
mkdir /data/subdir
echo "Nested file" > /data/subdir/file3.txt

# 현재 상태 확인
ls -la /data
ls -la /data/subdir

exit
```

### 2-3. 컨테이너 삭제 후 같은 볼륨으로 새 컨테이너 생성

```bash
# 첫 번째 컨테이너 삭제
docker rm vol-test

# 같은 볼륨으로 새 컨테이너 실행
docker run -it --name vol-test2 -v mydata:/data skilleatlab.azurecr.io/lab/busybox:latest
```

컨테이너 내부에서:

```bash
# 이전에 만든 파일이 여전히 존재!
ls -la /data
cat /data/file1.txt
cat /data/subdir/file3.txt

# 새 파일도 추가 가능
echo "New data from container 2" >> /data/file1.txt
cat /data/file1.txt

exit
```

### 2-4. 여러 컨테이너가 같은 볼륨 공유하기

> vol-test2가 아직 실행 중인 상태에서 진행합니다.
> **새 터미널을 열고 VM에 SSH 접속**한 뒤 아래 명령어를 실행하세요.

```bash
docker run -it --name vol-test3 -v mydata:/data skilleatlab.azurecr.io/lab/busybox:latest
```

컨테이너 내부에서:

```bash
# 다른 컨테이너가 작성한 파일도 볼 수 있음
cat /data/file1.txt  # 컨테이너 2에서 수정된 내용 포함

# 이 컨테이너에서도 수정
echo "Updated by container 3" > /data/file1.txt

exit
```

정리:

```bash
docker rm vol-test2 vol-test3
docker volume rm mydata
```

**핵심 학습:**

- Volume은 컨테이너 삭제 후에도 데이터 유지
- 여러 컨테이너가 같은 볼륨을 공유 가능
- 도커가 자동으로 관리하므로 편리함

## 3) Bind Mount 실습

### 3-1. 호스트 디렉토리 준비

```bash
# 호스트에서 실행
mkdir -p ~/bind-test/config
mkdir -p ~/bind-test/logs

# 초기 파일 생성
echo "host_config=production" > ~/bind-test/config/app.conf
echo "2026-04-28 App started" > ~/bind-test/logs/app.log

# 호스트에서 확인
ls -la ~/bind-test
cat ~/bind-test/config/app.conf
```

### 3-2. Bind Mount로 컨테이너 실행

```bash
docker run -it --name bind-test -v ~/bind-test:/app skilleatlab.azurecr.io/lab/busybox:latest
```

컨테이너 내부에서:

```bash
# 호스트의 파일이 보임
ls -la /app
cat /app/config/app.conf
cat /app/logs/app.log

# 호스트의 파일을 컨테이너에서 수정
echo "Updated by container" >> /app/config/app.conf

# 새로운 파일 생성
echo "2024-01-17 Processing complete" >> /app/logs/app.log
echo "process_status=completed" > /app/status.txt

exit
```

### 3-3. 호스트에서 변경사항 확인

```bash
# 컨테이너에서 수정한 파일이 호스트에 반영됨
cat ~/bind-test/config/app.conf
cat ~/bind-test/logs/app.log
cat ~/bind-test/status.txt

# 호스트에서 새 파일 생성
echo "host_added_data=yes" > ~/bind-test/new_from_host.txt
```

### 3-4. 같은 Bind Mount로 새 컨테이너 실행

```bash
docker run -it --name bind-test2 -v ~/bind-test:/app skilleatlab.azurecr.io/lab/busybox:latest
```

컨테이너 내부에서:

```bash
# 호스트에서 추가한 파일도 보임 (실시간 동기화)
ls -la /app
cat /app/new_from_host.txt

exit
```

정리:

```bash
docker rm bind-test bind-test2
```

**핵심 학습:**

- Bind Mount: 호스트 경로를 직접 컨테이너에 연결
- 양방향 실시간 동기화
- 개발 환경에서 코드 수정 시 유용 (핫 리로드)

## 4) tmpfs 볼륨 실습

### 4-1. tmpfs 마운트로 컨테이너 실행

```bash
docker run -it --name tmpfs-test --tmpfs /app:rw,size=64m skilleatlab.azurecr.io/lab/busybox:latest
```

컨테이너 내부에서:

```bash
# 메모리 기반 저장소에 파일 생성
mkdir /app/temp
echo "Temporary cache data" > /app/temp/cache.dat
echo "Session info" > /app/temp/session.txt

# 파일 확인
ls -la /app/temp
cat /app/temp/cache.dat

# 메모리 사용량 확인
df -h /app

exit
```

### 4-2. 컨테이너 재시작 후 데이터 확인

```bash
# 컨테이너 재시작
docker restart tmpfs-test

# 재시작된 컨테이너에 접속
docker exec -it tmpfs-test sh
```

컨테이너 내부에서:

```bash
# tmpfs의 데이터가 모두 사라짐 (메모리 기반이므로)
ls -la /app  # 디렉토리 존재하지만 비어있음

exit
```

### 4-3. 일반 볼륨과 tmpfs 함께 사용

```bash
docker volume create persistent-data

docker run -it \
  --name multi-test \
  -v persistent-data:/persistent \
  --tmpfs /temp:rw,size=64m \
  skilleatlab.azurecr.io/lab/busybox:latest
```

컨테이너 내부에서:

```bash
# 일반 볼륨에 저장 (재시작 후 유지)
echo "Persistent data" > /persistent/data.txt

# tmpfs에 저장 (재시작 후 삭제)
echo "Temporary data" > /temp/cache.txt

ls -la /persistent
ls -la /temp

exit
```

재시작 후 확인:

```bash
docker restart multi-test
docker exec -it multi-test sh
```

컨테이너 내부에서:

```bash
# Persistent 데이터는 유지
cat /persistent/data.txt

# Temp 데이터는 사라짐
ls /temp  # 비어있음

exit
```

정리:

```bash
docker rm tmpfs-test multi-test
docker volume rm persistent-data
```

**핵심 학습:**

- tmpfs: 메모리 기반, 재시작 시 삭제
- 임시 캐시나 세션 정보에 유용
- 일반 볼륨과 함께 사용 가능

## 5) 정리 및 상황별 선택 기준

### 5-1. 모든 컨테이너 및 볼륨 정리

```bash
# 실행 중인 모든 컨테이너 중지
docker stop $(docker ps -q) 2>/dev/null

# 모든 컨테이너 삭제
docker rm $(docker ps -aq) 2>/dev/null

# 사용 중이지 않은 볼륨 삭제
docker volume prune -f

# 정리 확인
docker ps -a
docker volume ls
```

### 5-2. 상황별 선택 기준

| 상황 | 추천 방식 | 이유 |
|------|---------|------|
| **데이터 영구 보존** | Volume | 도커 관리, 백업 용이 |
| **개발 중 코드 수정** | Bind Mount | 실시간 동기화, 핫 리로드 |
| **임시 캐시/세션** | tmpfs | 빠른 성능, 메모리 활용 |
| **여러 컨테이너 공유** | Volume | 안정적, 접근 제어 |
| **호스트 특정 경로 사용** | Bind Mount | 호스트 경로 직접 연결 |

---

## 📝 핵심 요약

1. **Ephemeral 특성**: 컨테이너 삭제 시 기본적으로 모든 데이터 손실
2. **3가지 볼륨**:
   - Volume: 도커 관리, 프로덕션 권장
   - Bind Mount: 호스트 경로, 개발 권장
   - tmpfs: 메모리 기반, 임시 데이터용
3. **선택 기준**: 데이터 보존 필요성과 사용 환경에 따라 결정
