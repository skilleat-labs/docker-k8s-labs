# 1-6. 도커 네트워크 (Docker Network) 실습

## 실습 목표

- Docker 네트워크 드라이버의 종류와 특징 이해하기
- 포트 바인딩과 사용자 정의 네트워크 생성/관리 실습하기
- 컨테이너 간 통신과 네트워크 트러블슈팅 체험하기

## 예상 소요 시간 (약 45분)

- 0~8분: 네트워크 개요와 기본 네트워크 확인
- 8~18분: bridge 네트워크 한계 체험과 사용자 정의 네트워크 생성
- 18~28분: 포트 바인딩 실습과 컨테이너 간 통신
- 28~38분: overlay 네트워크와 고급 기능
- 38~45분: 트러블슈팅 실습 및 정리

## 사전 조건

- Docker 설치 완료
- 기본 컨테이너 실행 경험

확인:

```bash
docker --version
docker network ls
```

## 1) 도커 네트워크 개요 체험

### 1-1. 기본 네트워크 확인

Docker 설치 시 자동 생성되는 3개 네트워크를 확인합니다.

```bash
docker network ls
```

관찰 포인트:

- `bridge`: 기본 브리지 네트워크
- `host`: 호스트 네트워크 직접 사용
- `none`: 네트워크 격리

### 1-2. 네트워크 드라이버 종류 이해

Docker 네트워크 드라이버:

- **bridge**: 컨테이너 간 격리된 네트워크 (기본)
- **host**: 호스트 네트워크 공유 (성능 좋음, 보안 낮음)
- **none**: 네트워크 연결 없음 (완전 격리)
- **overlay**: 여러 호스트에 걸친 컨테이너 통신을 위한 네트워크 (Swarm/K8s 환경에서만 사용 가능)
- **macvlan**: 컨테이너에 실제 MAC 주소를 부여해 물리 네트워크 장비처럼 동작하게 하는 네트워크 (Linux 서버 환경 필요)

## 2) bridge 네트워크 한계 체험

### 2-1. 기본 bridge 네트워크 사용

```bash
# 기본 bridge 네트워크로 컨테이너 실행
docker run -d --name web1 -p 8081:80 nginx:alpine
docker run -d --name web2 -p 8082:80 nginx:alpine

# 컨테이너 IP 확인
docker inspect web1 | grep -A 10 "Networks"
docker inspect web2 | grep -A 10 "Networks"
```

### 2-2. 컨테이너 간 통신 시도 (기본 bridge 한계)

```bash
# web1에서 web2로 ping 시도 (이름으로)
docker exec web1 ping -c 3 web2

# 실패하는 이유: 기본 bridge는 이름으로 통신 불가 (DNS 해석 안됨)

# IP로 직접 시도 (위 2-1에서 확인한 web2의 IP 사용)
docker exec web1 ping -c 3 <web2_IP>

# IP로는 성공 — 기본 bridge는 DNS가 없을 뿐, IP 통신 자체는 가능
```

관찰 포인트:

- 이름(web2)으로 ping → 실패 (DNS 해석 불가)
- IP로 ping → 성공 (IP 통신은 가능)
- 기본 bridge는 IP 통신만 가능, 이름으로 통신 불가
- 보안과 격리에는 좋지만 유연성이 떨어짐

### 2-3. 기본 bridge의 문제점

- 컨테이너 이름으로 통신 불가 (IP만 가능)
- 모든 컨테이너가 같은 네트워크에 있어 보안 취약
- 재시작 시 IP 변경으로 통신 끊김 가능성

## 3) host와 none 네트워크 실습

### 3-1. host 네트워크 체험

!!! warning "사전 점검: 호스트 80 포트 확인"
    host 네트워크는 컨테이너가 호스트 포트를 직접 사용합니다.
    호스트에 nginx 등이 이미 80 포트를 점유하고 있으면 컨테이너가 즉시 종료됩니다.

    ```bash
    # 80 포트 사용 여부 확인
    ss -tlnp | grep :80

    # 호스트 nginx가 실행 중이면 중지
    sudo systemctl stop nginx
    ```

```bash
# host 네트워크로 컨테이너 실행
docker run -d --name host-test --network host nginx:alpine

# 호스트에서 직접 접근 가능 (포트 바인딩 불필요)
curl http://localhost

# 컨테이너 내부 네트워크 확인
docker exec host-test ip addr
```

관찰 포인트:

- 호스트의 네트워크 인터페이스 직접 사용
- 최고 성능, 최저 보안
- 포트 충돌 가능성
- **포트 변경 불가**: `-p` 옵션이 무시됨 — 컨테이너가 호스트 포트를 그대로 사용하므로 포트 매핑 자체가 의미 없음

### 3-2. none 네트워크 체험

```bash
# 네트워크 없는 컨테이너 실행
docker run -d --name none-test --network none alpine:latest sleep 300

# 네트워크 인터페이스 확인
docker exec none-test ip addr

# 외부 통신 불가 확인
docker exec none-test ping -c 3 8.8.8.8
```

관찰 포인트:

- 완전 격리된 환경
- 보안이 중요한 컨테이너에 유용
- 외부 통신 전혀 불가

### 3-3. 정리

```bash
docker rm -f host-test none-test
```

**사용 시나리오:**

- **host**: 고성능이 필요한 경우 (모니터링, 로깅)
- **none**: 보안이 최우선인 경우 (암호화 작업)

## 4) 포트 바인딩 완전 실습

### 4-1. -p 옵션 문법 이해

```bash
# 기본 포트 바인딩
docker run -d --name port-test1 -p 8083:80 nginx:alpine

# 특정 IP 바인딩 (localhost만)
docker run -d --name port-test2 -p 127.0.0.1:8084:80 nginx:alpine

# 확인
curl http://localhost:8083  # 성공
curl http://localhost:8084  # 성공
curl http://192.168.1.100:8084  # 실패 (localhost만 바인딩)
```

### 4-2. -P 옵션 (자동 포트 할당)

```bash
# EXPOSE된 모든 포트를 자동 바인딩
docker run -d --name port-auto -P nginx:alpine

# 할당된 포트 확인
docker port port-auto
```

### 4-3. EXPOSE 명령어의 역할

Dockerfile에서 EXPOSE는 문서화 역할만 함:

```bash
# EXPOSE 없이도 포트 바인딩 가능
docker run -d --name no-expose -p 8085:80 alpine:latest \
  sh -c "echo 'Hello' > /tmp/index.html && httpd -p 80 -h /tmp"

curl http://localhost:8085
```

관찰 포인트:

- EXPOSE는 강제사항 아님, 문서화 용도
- -p로 호스트 포트 지정 필수

## 5) 사용자 정의 네트워크 생성/관리

### 5-1. 사용자 정의 bridge 네트워크 생성

```bash
# 사용자 정의 네트워크 생성
docker network create --driver bridge my-network

# 네트워크 확인
docker network ls
docker network inspect my-network
```

### 5-2. 사용자 정의 네트워크로 컨테이너 실행

```bash
# 같은 네트워크에 컨테이너 실행
docker run -d --name app1 --network my-network nginx:alpine
docker run -d --name app2 --network my-network nginx:alpine

# IP 주소 확인
docker inspect app1 | grep -A 5 "Networks"
docker inspect app2 | grep -A 5 "Networks"
```

### 5-3. 컨테이너 간 이름으로 통신

```bash
# app1에서 app2로 ping (이름으로 가능!)
docker exec app1 ping -c 3 app2

# HTTP 통신도 가능
docker exec app1 curl http://app2
```

관찰 포인트:

- 사용자 정의 네트워크에서는 DNS 해석 가능
- 컨테이너 이름으로 직접 통신

### 5-4. 네트워크 연결/해제

```bash
# 기존 컨테이너를 네트워크에 연결
docker network connect my-network port-test1

# 연결된 컨테이너 확인
docker network inspect my-network

# 통신 확인
docker exec app1 curl http://port-test1

# 네트워크에서 연결 해제
docker network disconnect my-network port-test1
```

### 5-5. 네트워크 삭제

```bash
# 컨테이너 중지 후 네트워크 삭제
docker rm -f app1 app2 port-test1 port-test2 port-auto no-expose
docker network rm my-network
```

## 6) 컨테이너 간 통신 구조 심화

### 6-1. 내부 DNS와 서비스 디스커버리

```bash
# 사용자 정의 네트워크 재생성
docker network create my-network

# 여러 컨테이너 실행
docker run -d --name web --network my-network nginx:alpine
docker run -d --name api --network my-network nginx:alpine
docker run -d --name db --network my-network alpine:latest sleep 300

# DNS 확인
docker exec web nslookup api
docker exec web nslookup db

# 통신 테스트
docker exec web curl http://api
docker exec api ping -c 2 db
```

### 6-2. 네트워크 격리 확인

```bash
# 다른 네트워크의 컨테이너 생성
docker network create isolated-network
docker run -d --name isolated --network isolated-network nginx:alpine

# 격리된 네트워크 간 통신 불가
docker exec web curl http://isolated  # 실패
```

### 6-3. --link 옵션 (deprecated) 비교

```bash
# link 옵션으로 컨테이너 연결 (권장하지 않음)
docker run -d --name legacy --link web:webserver nginx:alpine

# link로 연결된 경우 DNS 확인
docker exec legacy ping -c 2 webserver
```

관찰 포인트:

- --link는 단방향 연결, 복잡한 구성에 부적합
- 사용자 정의 네트워크 권장

## 7) overlay 네트워크와 고급 기능

### 7-1. overlay 네트워크 생성 (Swarm 전제)

```bash
# Docker Swarm 초기화 (단일 노드)
docker swarm init

# overlay 네트워크 생성
docker network create --driver overlay overlay-net

# Swarm 서비스로 컨테이너 실행
docker service create --name overlay-web --network overlay-net --replicas 2 nginx:alpine

# 서비스 확인
docker service ls
docker service ps overlay-web
```

### 7-2. K8s 연관성 이해

overlay 네트워크는 K8s의 Service 개념과 유사:

- 여러 노드에 걸친 컨테이너 간 통신
- 로드 밸런싱
- 서비스 디스커버리

### 7-3. Swarm 정리

```bash
docker service rm overlay-web
docker network rm overlay-net
docker swarm leave --force
```

## 8) 네트워크 트러블슈팅 실전

### 8-1. 1단계: 네트워크 상태 확인

```bash
# 모든 네트워크 확인
docker network ls

# 특정 네트워크 상세 정보
docker network inspect my-network

# 컨테이너의 네트워크 정보
docker inspect web | grep -A 20 "NetworkSettings"
```

### 8-2. 2단계: 컨테이너 상태 점검

```bash
# 컨테이너 실행 상태
docker ps -a

# 컨테이너 로그 확인
docker logs web

# 컨테이너 내부 네트워크 도구 사용
docker exec web ip addr
docker exec web ip route
docker exec web cat /etc/resolv.conf
```

### 8-3. 3단계: 통신 테스트와 로그 분석

```bash
# DNS 해석 테스트
docker exec web nslookup api

# 포트 연결 테스트
docker exec web nc -zv api 80

# 호스트에서 컨테이너 포트 확인
docker port web

# iptables 규칙 확인 (고급)
docker exec web iptables -L
```

### 8-4. 일반적인 문제 해결

**문제 1: 컨테이너 간 통신 안됨**

```bash
# 해결: 같은 사용자 정의 네트워크 사용
docker network connect my-network container-name
```

**문제 2: 포트 바인딩 실패**

```bash
# 해결: 포트 충돌 확인
netstat -tulpn | grep :8080
docker ps | grep 8080
```

**문제 3: DNS 해석 실패**

```bash
# 해결: 네트워크 재연결 또는 재시작
docker network disconnect my-network container-name
docker network connect my-network container-name
```

## 9) 정리 및 핵심 학습

### 9-1. 전체 정리

```bash
# 모든 컨테이너 중지/삭제
docker rm -f $(docker ps -aq) 2>/dev/null || true

# 사용자 정의 네트워크 삭제
docker network rm my-network isolated-network 2>/dev/null || true

# 네트워크 목록 확인
docker network ls
```

### 9-2. 네트워크 선택 가이드

| 네트워크 타입 | 사용 시기 | 장점 | 단점 |
|---------------|-----------|------|------|
| **기본 bridge** | 간단한 단일 호스트 | 자동 생성 | 이름 통신 불가 |
| **사용자 정의 bridge** | 다중 컨테이너 앱 | DNS 지원, 격리 | 설정 필요 |
| **host** | 최고 성능 필요 | 빠름 | 보안 낮음 |
| **none** | 완전 격리 | 보안 높음 | 통신 불가 |
| **overlay** | Swarm/K8s | 멀티 호스트 | Swarm 필요 |

## 📝 핵심 요약

1. **드라이버 이해**: bridge(기본), host(성능), none(격리), overlay(멀티호스트) 네트워크 드라이버의 특징과 사용 시나리오
2. **포트 바인딩 + 사용자 정의 네트워크**: -p 문법으로 호스트 포트 매핑, 사용자 정의 네트워크로 DNS 지원 컨테이너 간 통신
3. **실습 명령어**: docker network create/ls/inspect, --network 옵션, docker exec로 내부 네트워크 도구 활용
