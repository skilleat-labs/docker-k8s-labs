# 1-3. Docker Hub와 이미지 실습

## 실습 목표

- Docker Hub에서 다양한 이미지를 직접 가져오기
- 가져온 이미지를 컨테이너로 실행해보기
- 이미지 캐시(레이어 재사용)가 무엇인지 실습으로 이해하기

## 예상 소요 시간 (약 30분)

- 0~8분: Docker Hub 이미지 탐색 및 pull
- 8~20분: 이미지별 컨테이너 실행
- 20~28분: 캐시/레이어 재사용 체감 실습
- 28~30분: 정리

## 사전 조건

- Docker 설치 완료
- 인터넷 연결 가능

확인:

```bash
docker --version
```

## 1) Docker Hub에서 이미지 찾기

Docker Hub: [https://hub.docker.com/](https://hub.docker.com/)

터미널 검색 예시:

```bash
docker search nginx
docker search redis
```

포인트:

- `OFFICIAL` 표시가 있으면 공식 이미지
- `TAG`를 명시하면 버전 고정 가능

## 2) 이미지 pull 실습

먼저 6개 이미지를 가져옵니다.

```bash
docker pull nginx:latest
docker pull httpd:latest
docker pull redis:latest
docker pull alpine:latest
docker pull python:3.11-slim
docker pull python:3.12-slim
```

목록 확인:

```bash
docker images
```

관찰 포인트:

- 이미지마다 `IMAGE ID`, `SIZE`가 다름
- 동일 계열 이미지끼리는 레이어를 공유할 수 있음

## 3) 이미지로 컨테이너 실행 실습

### 3-1. Nginx 실행

```bash
docker run -d --name nginx-web -p 8081:80 nginx:latest
curl -I http://localhost:8081
```

### 3-2. Apache(httpd) 실행

```bash
docker run -d --name httpd-web -p 8082:80 httpd:latest
curl -I http://localhost:8082
```

### 3-3. Redis 실행

```bash
docker run -d --name redis-db -p 6379:6379 redis:latest
docker logs redis-db | head
```

### 3-4. Alpine 실행 (짧은 테스트용)

> **Alpine이란?**
> Alpine Linux는 약 5MB짜리 초경량 리눅스 OS 이미지입니다.
> nginx처럼 특정 서비스가 내장된 것도 아니고, bash만 달랑 있는 것도 아닙니다.
> 기본 shell(`sh`)과 핵심 유틸리티만 담긴 **최소한의 리눅스 환경**으로,
> 간단한 명령 실행 테스트나 다른 이미지의 베이스로 자주 사용됩니다.

```bash
docker run --name alpine-test alpine:latest echo "hello from alpine"
```

상태 확인:

```bash
docker ps
docker ps -a
```

## 4) 컨테이너는 "프로세스"라는 점 이해하기

핵심: 컨테이너는 내부의 메인 프로세스(PID 1)가 끝나면 같이 종료됩니다.

### 4-1. 바로 종료되는 컨테이너 보기

```bash
docker run --name exit-now alpine:latest echo "bye"
docker ps -a | grep exit-now
```

관찰 포인트:

- `echo` 명령은 바로 끝남
- 메인 프로세스가 끝났기 때문에 컨테이너 상태가 `Exited`로 바뀜

### 4-2. 살아있게 유지하기 (`sleep`)

```bash
docker run -d --name keep-alive alpine:latest sleep 600
docker ps | grep keep-alive
```

관찰 포인트:

- `sleep 600` 프로세스가 600초 동안 실행됨
- 그동안 컨테이너도 `Up` 상태로 유지됨

### 4-3. 종료/재시작으로 동작 확인

```bash
docker stop keep-alive
docker ps -a | grep keep-alive
docker start keep-alive
docker ps | grep keep-alive
```

### 4-4. 무한 유지 패턴도 확인

```bash
docker run -d --name keep-forever alpine:latest sh -c "while true; do sleep 60; done"
docker ps | grep keep-forever
```

정리:

- 컨테이너는 VM처럼 "항상 켜져 있는 OS"가 아님
- 메인 프로세스를 무엇으로 주느냐에 따라 바로 종료되거나 계속 실행됨

## 5) 캐시(레이어 재사용) 체감 실습

> **참고: Docker 버전에 따라 출력이 다릅니다**
>
> Docker Engine v29(2025년 11월) 이후 신규 설치, Docker Desktop 4.34(2024년 8월) 이후 신규 설치는
> 기본 이미지 저장소가 **containerd image store**로 바뀌었습니다.
>
> 이전 방식(graphdriver)에서는 이미 로컬에 있는 레이어가 `Already exists`로 표시됐지만,
> 새 방식(containerd)에서는 **해당 레이어 라인이 출력에서 아예 사라집니다.**
> 레이어 재사용은 동일하게 일어나지만, 눈에 보이는 출력이 달라진 것입니다.
>
> 내 환경 확인:
> ```bash
> docker info | grep "Storage Driver"
> ```
> - `overlay2` → graphdriver (이전 방식, `Already exists` 보임)
> - `overlayfs` → containerd image store (새 방식, 라인 자체가 사라짐)

### 5-1. 같은 이미지를 다시 pull

이미 받은 이미지를 다시 pull하면 레이어를 재다운로드하지 않습니다.

```bash
docker pull nginx:latest
```

**예상 출력 (이미 로컬에 있을 때):**

```
latest: Pulling from library/nginx
Digest: sha256:...
Status: Image is up to date for nginx:latest
```

관찰 포인트:

- `Image is up to date` → 레이어를 전혀 다운로드하지 않음
- 네트워크 트래픽 없이 즉시 완료됨

### 5-2. 레이어 재사용을 직접 눈으로 확인하기

pull 출력만으로는 재사용 여부가 잘 안 보이므로, `docker image inspect`로 레이어 해시를 직접 비교합니다.

같은 Debian 베이스를 쓰는 두 Python 이미지를 받습니다:

```bash
docker pull python:3.11-slim
docker pull python:3.12-slim
```

레이어 목록 비교:

```bash
docker image inspect python:3.11-slim | jq '.[0].RootFS.Layers'
docker image inspect python:3.12-slim | jq '.[0].RootFS.Layers'
```

**예상 출력:**

```
// python:3.11-slim
[
  "sha256:e6133723...",   ← Debian 베이스 레이어 (3.12와 동일)
  "sha256:384ef01e...",   ← Python 3.11 전용
  "sha256:2eeb7784...",   ← Python 3.11 전용
  "sha256:1f339219..."    ← Python 3.11 전용
]

// python:3.12-slim
[
  "sha256:e6133723...",   ← Debian 베이스 레이어 (3.11과 동일) ← 이 부분!
  "sha256:c24d0821...",   ← Python 3.12 전용
  "sha256:8d5257b6...",   ← Python 3.12 전용
  "sha256:e85497ba..."    ← Python 3.12 전용
]
```

관찰 포인트:

- 첫 번째 해시(`e6133723...`)가 **두 이미지에서 완전히 동일** → Debian OS 베이스 레이어를 공유
- 나머지 3개는 Python 버전마다 다름 → 언어 런타임이 달라지는 부분
- 즉, **두 이미지를 받아도 공유 레이어는 디스크에 한 번만 저장됨**
- pull 출력에서는 이 공유 여부가 보이지 않지만, inspect로 실제 레이어 해시를 보면 증명됨

### 5-3. 같은 이미지로 새 컨테이너 추가 실행

```bash
docker run -d --name nginx-web-2 -p 8083:80 nginx:latest
docker ps
```

관찰 포인트:

- `nginx:latest` 이미지는 한 번 받아두면 재사용됨
- 새 컨테이너는 이미지를 복제 다운로드하는 게 아니라 기존 로컬 이미지 레이어를 기반으로 빠르게 생성됨

### 5-4. 캐시가 깨지는 경우

아래 중 하나면 모든 레이어를 새로 다운로드합니다.

- 로컬 이미지와 레이어를 모두 삭제한 경우 (`docker system prune`)
- 원격 이미지가 업데이트되어 digest(내용)가 바뀐 경우
- 전혀 다른 계열의 이미지 pull (예: `redis:latest`)

## 6) 정리: 캐시된다는 뜻

"이미지가 캐시된다"는 뜻은 아래와 같습니다.

- 레이어 단위로 로컬 디스크에 저장됨
- 동일 레이어는 재다운로드하지 않음
- 같은 이미지를 여러 컨테이너가 공유 기반으로 사용 가능
- 결과적으로 pull/실행 속도와 네트워크 사용량이 개선됨

## 7) 실습 정리(삭제)

```bash
docker rm -f nginx-web nginx-web-2 httpd-web redis-db alpine-test 2>/dev/null || true
docker rm -f exit-now keep-alive keep-forever 2>/dev/null || true
```

이미지까지 지우고 싶다면:

```bash
docker rmi nginx:latest httpd:latest redis:latest alpine:latest
docker rmi python:3.11-slim python:3.12-slim
```

## 8) Azure Container Registry (ACR)로 이미지 push하기

### 8-1. Azure 계정 준비

각 팀별 계정 정보:

| 팀 | 이메일 | 비밀번호 |
|-----|--------|---------|
| Team 01 | `test01@nrkim0615outlook.onmicrosoft.com` | 강사 제공 |
| Team 02 | `test02@nrkim0615outlook.onmicrosoft.com` | 강사 제공 |
| Team 03 | `test03@nrkim0615outlook.onmicrosoft.com` | 강사 제공 |

### 8-2. Azure CLI 로그인

```bash
az login
```

이메일과 비밀번호를 입력하여 로그인합니다.

로그인 확인:

```bash
az account show
```

### 8-3. Azure Container Registry에 로그인

```bash
az acr login --name skillitacr
```

확인:

```bash
docker login skillitacr.azurecr.io
```

### 8-4. 로컬 이미지에 태그 추가

로컬의 `nginx:latest` 이미지를 Azure ACR로 push하기 위해 태그를 변경합니다.

```bash
# 현재 로컬 이미지 확인
docker images | grep nginx

# ACR 레지스트리 주소로 태그 변경
docker tag nginx:latest skillitacr.azurecr.io/myteam/nginx:v1.0

# 태그 확인
docker images | grep skillitacr
```

포인트:

- `skillitacr.azurecr.io`: Azure Container Registry 주소
- `myteam`: 저장소(repository) 이름 (팀 이름으로 변경 가능)
- `v1.0`: 버전 태그

### 8-5. Azure ACR로 push

```bash
docker push skillitacr.azurecr.io/myteam/nginx:v1.0
```

관찰 포인트:

- 로컬 이미지의 레이어를 ACR로 업로드
- 처음 push는 시간이 걸릴 수 있음
- 완료 후 "Pushed" 메시지 확인

### 8-6. Azure ACR에 저장된 이미지 확인

```bash
az acr repository list --name skillitacr --output table
```

특정 저장소의 이미지 태그 확인:

```bash
az acr repository show-tags --name skillitacr --repository myteam/nginx
```

### 8-7. Azure ACR에서 pull 및 실행

다른 환경이나 팀원 컴퓨터에서 ACR의 이미지를 받아와 실행할 수 있습니다.

```bash
docker pull skillitacr.azurecr.io/myteam/nginx:v1.0

docker run -d --name acr-nginx -p 8084:80 skillitacr.azurecr.io/myteam/nginx:v1.0

curl -I http://localhost:8084
```

### 8-8. 정리

```bash
docker rm -f acr-nginx 2>/dev/null || true
```

**핵심 학습:**

- Docker Hub는 공개 레지스트리, Azure ACR은 비공개/기업용 레지스트리
- 이미지에 태그를 추가하여 다양한 레지스트리로 push/pull 가능
- 팀 전체가 같은 ACR을 사용하면 이미지 공유가 용이함
