- 하드코딩된 설정의 문제점을 직접 체험하기

- Dockerfile의 `ARG`와 `ENV` 차이를 실습으로 확인하기

## 사용 이미지

| 태그 | 설명 |
|------|------|
| `skilleatlab.azurecr.io/env-lab:hardcoded` | DB 비밀번호가 이미지에 하드코딩된 버전 |
| `skilleatlab.azurecr.io/env-lab:v1` | 환경변수로 설정을 주입받는 개선 버전 |
| `skilleatlab.azurecr.io/env-lab:arg-demo` | ARG vs ENV 차이 시연용 |

세 이미지 모두 **환경변수 값을 웹 페이지에 표시하는 Flask 앱**으로, Dockerfile만 다르게 만들어졌습니다.

### `hardcoded` — 비밀번호가 이미지에 그대로 박혀 있음

```dockerfile
ENV APP_ENV=production
ENV DB_HOST=prod-db.internal
ENV DB_PORT=5432
ENV DB_PASSWORD=super_secret_1234   # ← 이미지 레이어에 영구적으로 남음
```

이 이미지를 `docker inspect`하면 누구나 비밀번호를 볼 수 있습니다.

### `v1` — 값 없이 정의만, 실행 시 주입

```dockerfile
ENV APP_ENV=development
ENV DB_HOST=localhost
ENV DB_PORT=5432
ENV DB_PASSWORD=                    # ← 빈 값, -e 또는 env_file로 채움
```

이미지 자체에는 비밀번호가 없고, 실행할 때 `-e DB_PASSWORD=...`로 주입합니다.

### `arg-demo` — ARG로 받아 ENV로 저장

```dockerfile
ARG BUILD_VERSION=unknown   # 빌드 시에만 존재, 이미지에 남지 않음
ARG BUILD_DATE=unknown

ENV APP_VERSION=$BUILD_VERSION   # ARG 값을 ENV로 복사 → 이미지에 남음
ENV BUILD_DATE=$BUILD_DATE
```

`ARG`는 빌드가 끝나면 사라지고, `ENV`로 옮긴 값만 런타임에 남습니다.

## 1) 문제 인식: 하드코딩의 위험성

### 1-1. 하드코딩된 이미지 실행

```bash
docker run -d --name hardcoded-app -p 5001:5000 \
  skilleatlab.azurecr.io/env-lab:hardcoded
```

브라우저에서 확인합니다.

```
http://192.168.56.10:5001
```

`DB_PASSWORD` 값이 화면에 그대로 출력되는 것을 확인합니다.

### 1-2. 이미지 레이어에서 비밀번호 확인

이미지를 inspect해서 ENV에 비밀번호가 박혀있는지 확인합니다.

```bash
docker inspect skilleatlab.azurecr.io/env-lab:hardcoded | grep -A 20 '"Env"'
```

관찰 포인트:

- `DB_PASSWORD=super_secret_1234` 가 이미지 메타데이터에 그대로 노출됨
- 이 이미지를 Docker Hub에 public으로 올리면 누구나 비밀번호를 볼 수 있음
- 이미지를 배포하는 순간 비밀번호도 함께 배포됨

### 1-3. 정리

```bash
docker rm -f hardcoded-app
```

## 2) 1차 개선: 환경변수로 주입

### 2-1. 개선된 이미지 실행 (기본값)

```bash
docker run -d --name env-app -p 5001:5000 \
  skilleatlab.azurecr.io/env-lab:v1
```

브라우저에서 확인하면 `DB_PASSWORD`가 비어있습니다.

```
http://192.168.56.10:5001
```

### 2-2. -e 옵션으로 값 주입

컨테이너를 다시 실행하면서 값을 주입합니다.

```bash
docker rm -f env-app

docker run -d --name env-app -p 5001:5000 \
  -e APP_ENV=production \
  -e DB_HOST=prod-db.internal \
  -e DB_PORT=5432 \
  -e DB_PASSWORD=my_secure_password \
  skilleatlab.azurecr.io/env-lab:v1
```

브라우저에서 2-1과 비교해보세요.

- 2-1: `DB_PASSWORD` → 비어있음
- 2-2: `DB_PASSWORD` → `my_secure_password`로 채워짐

**이미지는 그대로인데 `-e`로 주입한 값만 바뀐 것**을 확인합니다.

### 2-3. 같은 이미지로 개발/운영 환경 분리

2-2의 `env-app`은 그대로 두고, 추가로 실행합니다.

**개발 환경:**

```bash
docker run -d --name dev-app -p 5002:5000 \
  -e APP_ENV=development \
  -e DB_HOST=localhost \
  -e DB_PASSWORD=devpassword \
  skilleatlab.azurecr.io/env-lab:v1
```

**운영 환경:**

```bash
docker run -d --name prod-app -p 5003:5000 \
  -e APP_ENV=production \
  -e DB_HOST=prod-db.internal \
  -e DB_PASSWORD=prodpassword \
  skilleatlab.azurecr.io/env-lab:v1
```

브라우저에서 세 개를 나란히 열어서 비교해보세요.

```
http://192.168.56.10:5001  ← env-app (2-2)
http://192.168.56.10:5002  ← dev-app
http://192.168.56.10:5003  ← prod-app
```

관찰 포인트:

- **이미지는 동일**, 주입하는 값만 다름
- 한 번 빌드한 이미지를 개발/운영 어디서나 재사용 가능

### 2-4. 정리

```bash
docker rm -f env-app dev-app prod-app
```

## 3) Dockerfile: ARG vs ENV

`ARG`와 `ENV`는 둘 다 Dockerfile에서 변수를 다루지만 **동작 시점이 다릅니다.**

| 구분 | 사용 시점 | 이미지에 남는가 | 런타임 변경 가능 |
|------|----------|----------------|----------------|
| `ARG` | 빌드 시에만 | ❌ | ❌ |
| `ENV` | 빌드 + 런타임 | ✅ | ✅ (`-e`로 덮어쓰기 가능) |

### 3-1. ARG vs ENV 이미지 실행

```bash
docker run -d --name arg-demo -p 5001:5000 \
  skilleatlab.azurecr.io/env-lab:arg-demo
```

브라우저에서 `APP_VERSION`과 `BUILD_DATE`를 확인합니다.

```
http://192.168.56.10:5001
```

### 3-2. 이미지 inspect로 차이 확인

```bash
docker inspect skilleatlab.azurecr.io/env-lab:arg-demo | grep -A 20 '"Env"'
```

관찰 포인트:

이 이미지의 Dockerfile에서는 이렇게 선언했습니다.

```
ARG BUILD_VERSION=unknown  →  ENV APP_VERSION=$BUILD_VERSION  (이름 바꿔서 복사)
ARG BUILD_DATE=unknown     →  ENV BUILD_DATE=$BUILD_DATE      (같은 이름으로 복사)
```

따라서 inspect 결과에서:

- `APP_VERSION=1.0.0` → **보임** (ARG `BUILD_VERSION`을 ENV로 복사했으므로)
- `BUILD_DATE=2024-xx-xx` → **보임** (ARG `BUILD_DATE`를 ENV로 복사했으므로)
- `BUILD_VERSION` → **없음** (ARG로만 선언, ENV로 복사하지 않았으므로 이미지에 흔적 없음)

ARG는 ENV로 옮기지 않으면 빌드가 끝난 뒤 완전히 사라집니다.

### 3-3. --build-arg로 직접 빌드해서 확인

```bash
mkdir ~/arg-test && cd ~/arg-test

# Dockerfile 작성
cat > Dockerfile <<'EOF'
FROM alpine:latest
ARG BUILD_VERSION=unknown
ENV APP_VERSION=$BUILD_VERSION
RUN echo "빌드 중 ARG 값: $BUILD_VERSION"
CMD ["sh", "-c", "echo APP_VERSION=$APP_VERSION"]
EOF

# ARG 없이 빌드
docker build -t arg-test:no-arg .

# ARG 전달해서 빌드
docker build --build-arg BUILD_VERSION=2.0.0 -t arg-test:with-arg .

# 실행 비교
docker run --rm arg-test:no-arg
docker run --rm arg-test:with-arg

# inspect로 ENV 확인
docker inspect arg-test:with-arg | grep -A 10 '"Env"'
```

관찰 포인트:

- `BUILD_VERSION`(ARG)은 inspect 결과에 없음
- `APP_VERSION`(ENV)은 inspect 결과에 남아있음
- ARG를 ENV로 복사하지 않으면 런타임에서 사용 불가

### 3-4. 정리

```bash
docker rm -f arg-demo
docker rmi arg-test:no-arg arg-test:with-arg
```

## 4) Compose에서 환경변수 관리

### 4-1. 실습 디렉토리 생성

```bash
mkdir ~/env-compose-lab && cd ~/env-compose-lab
```

### 4-2. environment: 직접 작성 (1단계)

```bash
nano docker-compose.yml
```

```yaml
services:
  app:
    image: skilleatlab.azurecr.io/env-lab:v1
    ports:
      - "5001:5000"
    environment:
      APP_ENV: development
      DB_HOST: localhost
      DB_PORT: 5432
      DB_PASSWORD: devpassword123
```

```bash
docker compose up -d
```

브라우저에서 확인 후 중지합니다.

```bash
docker compose down
```

### 4-3. .env 파일로 분리 (2단계)

`docker-compose.yml`에 비밀번호가 그대로 있는 것이 불편합니다. `.env` 파일로 분리합니다.

```bash
nano .env
```

```
APP_ENV=development
DB_HOST=localhost
DB_PORT=5432
DB_PASSWORD=devpassword123
```

`docker-compose.yml`을 수정합니다.

```bash
nano docker-compose.yml
```

```yaml
services:
  app:
    image: skilleatlab.azurecr.io/env-lab:v1
    ports:
      - "5001:5000"
    env_file:
      - .env
```

```bash
docker compose up -d
```

브라우저에서 값이 동일하게 나오는지 확인합니다.

```bash
docker compose down
```

### 4-4. .gitignore 설정

`.env`에는 비밀번호가 있으므로 git에 올라가면 안 됩니다.

```bash
echo ".env" >> .gitignore
echo ".env.*" >> .gitignore
cat .gitignore
```

!!! warning ".env를 git에 올리면 안 되는 이유"
    `.env`에는 DB 비밀번호, API 키 등 민감정보가 담깁니다.
    git에 한 번 올라가면 삭제해도 히스토리에 남기 때문에
    **처음부터 .gitignore에 등록하는 것이 중요합니다.**

## 5) 개발/운영 환경 분리 전략

### 5-1. 환경별 .env 파일 작성

```bash
# 개발 환경
nano .env.dev
```

```
APP_ENV=development
DB_HOST=localhost
DB_PORT=5432
DB_PASSWORD=devpassword123
```

```bash
# 운영 환경
nano .env.prod
```

```
APP_ENV=production
DB_HOST=prod-db.internal
DB_PORT=5432
DB_PASSWORD=strongProdPassword!
```

### 5-2. --env-file 옵션으로 선택 실행

```bash
# 개발 환경으로 실행
docker compose --env-file .env.dev up -d
```

브라우저에서 `APP_ENV=development` 확인 후:

```bash
docker compose down
```

```bash
# 운영 환경으로 실행
docker compose --env-file .env.prod up -d
```

브라우저에서 `APP_ENV=production` 확인합니다.

관찰 포인트:

- `docker-compose.yml`은 변경 없이 동일
- `--env-file`만 바꾸면 개발/운영 환경 전환 가능
- 이미지도 동일, 설정만 다름

```bash
docker compose down
```

## 6) 민감정보 별도 관리

### 6-1. 일반 설정과 민감정보 파일 분리

운영에서는 일반 설정과 민감정보를 파일로 분리해서 관리합니다.

```bash
# 일반 설정
nano .env.prod
```

```
APP_ENV=production
DB_HOST=prod-db.internal
DB_PORT=5432
```

```bash
# 민감정보만 별도 파일
nano .env.secret
```

```
DB_PASSWORD=strongProdPassword!
```

`docker-compose.yml`에서 두 파일을 같이 사용합니다.

```yaml
services:
  app:
    image: skilleatlab.azurecr.io/env-lab:v1
    ports:
      - "5001:5000"
    env_file:
      - .env.prod
      - .env.secret
```

```bash
docker compose up -d
```

브라우저에서 두 파일의 값이 합쳐져서 표시되는지 확인합니다.

```bash
docker compose down
```

### 6-2. 관리 전략 정리

| 파일 | 내용 | git 포함 |
|------|------|---------|
| `docker-compose.yml` | 서비스 구조 정의 | ✅ |
| `.env.dev` | 개발 환경 설정 (비밀번호 제외) | ✅ (팀 공유) |
| `.env.prod` | 운영 환경 설정 (비밀번호 제외) | ✅ (팀 공유) |
| `.env.secret` | 비밀번호, API 키 등 민감정보 | ❌ (.gitignore) |

## 📝 핵심 요약

1. **하드코딩 금지**: 비밀번호, API 키는 이미지나 코드에 직접 넣지 않는다
2. **ENV로 주입**: 실행 시 `-e` 또는 `env_file:`로 값을 주입한다
3. **ARG vs ENV**: ARG는 빌드 시에만, ENV는 런타임까지 유지된다
4. **환경 분리**: `.env.dev` / `.env.prod`로 환경별 설정 파일을 분리한다
5. **민감정보**: `.env.secret`으로 별도 관리하고 반드시 `.gitignore`에 등록한다
