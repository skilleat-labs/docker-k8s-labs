# 2-3. ConfigMap & Secret 실습

## 실습 목표

- 환경변수를 코드에 직접 박는 방식의 문제를 체감한다.
- ConfigMap으로 비민감 설정을 분리 관리한다.
- Secret으로 민감 정보를 안전하게 주입한다.
- `envFrom`, `env.valueFrom`, 볼륨 마운트 3가지 주입 방식을 모두 실습한다.
- ConfigMap 변경이 실행 중인 Pod에 어떻게 반영되는지 확인한다.

---

## 왜 ConfigMap과 Secret이 필요한가?

먼저 문제를 직접 체감합니다.

### 환경변수를 YAML에 직접 넣는 방식

`pod-hardcoded.yaml` 파일을 만듭니다.

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: pod-hardcoded
spec:
  containers:
    - name: app
      image: busybox:1.36
      command: ["sh", "-c", "echo DB=$DB_HOST PASS=$DB_PASSWORD && sleep 3600"]
      env:
        - name: DB_HOST
          value: "mysql-svc"
        - name: DB_PASSWORD
          value: "supersecret123"    # 비밀번호가 YAML에 노출
  restartPolicy: Never
```

```bash
kubectl apply -f pod-hardcoded.yaml
kubectl logs pod-hardcoded
```

```
DB=mysql-svc PASS=supersecret123
```

이 방식의 문제점:

- 비밀번호가 YAML 파일에 평문으로 노출됩니다.
- 환경(개발/운영)이 바뀔 때마다 YAML을 수정하고 재배포해야 합니다.
- Git에 올리면 비밀번호가 히스토리에 영구히 남습니다.

ConfigMap과 Secret은 이 문제를 해결합니다. **설정과 코드를 분리**하여 YAML을 수정하지 않고 설정만 바꿀 수 있습니다.

```bash
kubectl delete pod pod-hardcoded
```

## 1) ConfigMap — 비민감 설정 분리

### ConfigMap이란?

DB 접속 주소, 포트, 환경 이름처럼 **노출돼도 괜찮은 설정값**을 별도 리소스로 분리합니다.

### 방법 A — kubectl 명령어로 생성

```bash
kubectl create configmap app-config \
  --from-literal=DB_HOST=mysql-svc \
  --from-literal=DB_PORT=3306 \
  --from-literal=APP_ENV=development
```

```bash
kubectl get configmaps
kubectl describe configmap app-config
```

`describe` 출력에서 `Data` 항목에 키-값이 평문으로 저장된 것을 확인합니다.

### 방법 B — YAML 파일로 생성

`configmap.yaml` 파일을 만듭니다.

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  DB_HOST: mysql-svc
  DB_PORT: "3306"
  APP_ENV: development
  app.properties: |
    log.level=INFO
    max.connections=100
    timeout.seconds=30
```

`data` 아래에는 단순 키-값뿐 아니라 `|`를 사용해 여러 줄 설정 파일 내용도 담을 수 있습니다.

```bash
kubectl apply -f configmap.yaml    # 이미 있으면 업데이트됨
kubectl get configmap app-config -o yaml
```

## 2) Secret — 민감 정보 분리

### Secret이란?

비밀번호, API 키, 인증서처럼 **노출되면 안 되는 값**을 저장합니다. 내부적으로 base64 인코딩되어 저장됩니다.

> base64는 **암호화가 아닙니다**. 누구나 디코딩할 수 있습니다. Secret의 보안은 RBAC(접근 권한 제어)으로 확보합니다.

### base64 인코딩 이해

```bash
echo -n "supersecret123" | base64    # 인코딩: c3VwZXJzZWNyZXQxMjM=
echo -n "c3VwZXJzZWNyZXQxMjM=" | base64 -d    # 디코딩: supersecret123
```

`-n` 옵션은 개행문자(`\n`)가 포함되지 않도록 합니다. 빠뜨리면 디코딩 시 값 뒤에 줄바꿈이 붙어 오류가 날 수 있습니다.

### 방법 A — kubectl 명령어 (자동 base64 인코딩)

```bash
kubectl create secret generic app-secret \
  --from-literal=DB_PASSWORD=supersecret123 \
  --from-literal=API_KEY=abc-def-ghi
```

```bash
kubectl get secrets
kubectl describe secret app-secret    # Values는 보이지 않음
```

### 방법 B — YAML 파일로 생성

`secret.yaml` 파일을 만듭니다. `data` 필드에는 base64 인코딩된 값을 넣습니다.

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: app-secret
type: Opaque
data:
  DB_PASSWORD: c3VwZXJzZWNyZXQxMjM=
  API_KEY: YWJjLWRlZi1naGk=
```

```bash
kubectl apply -f secret.yaml
```

### 방법 C — stringData 사용 (평문 작성, 자동 인코딩)

base64 변환이 번거로울 때 `stringData`를 사용합니다. 적용 시 K8s가 자동으로 base64 인코딩합니다.

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: app-secret
type: Opaque
stringData:
  DB_PASSWORD: supersecret123
  API_KEY: abc-def-ghi
```

### Secret 값 확인

```bash
kubectl get secret app-secret -o yaml    # base64 인코딩된 값 확인
kubectl get secret app-secret -o jsonpath='{.data.DB_PASSWORD}' | base64 -d
```

## 3) 주입 방식 1 — envFrom (전체 주입)

ConfigMap이나 Secret의 **모든 키를 한번에** 환경변수로 주입합니다.

`pod-envfrom.yaml` 파일을 만듭니다.

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: pod-envfrom
spec:
  containers:
    - name: app
      image: busybox:1.36
      command: ["sh", "-c", "env | sort && sleep 3600"]
      envFrom:
        - configMapRef:
            name: app-config
        - secretRef:
            name: app-secret
  restartPolicy: Never
```

```bash
kubectl apply -f pod-envfrom.yaml
kubectl logs pod-envfrom
```

출력에서 `DB_HOST`, `DB_PORT`, `APP_ENV`, `DB_PASSWORD`, `API_KEY`가 모두 환경변수로 주입된 것을 확인합니다.

Pod 내부에서 직접 확인:

```bash
kubectl exec pod-envfrom -- env | grep -E "DB_|APP_|API_"
```

## 4) 주입 방식 2 — env.valueFrom (선택 주입)

여러 ConfigMap·Secret 중 **특정 키만 골라서** 원하는 환경변수 이름으로 주입합니다.

`pod-valuefrom.yaml` 파일을 만듭니다.

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: pod-valuefrom
spec:
  containers:
    - name: app
      image: busybox:1.36
      command: ["sh", "-c", "echo HOST=$DB_HOST PORT=$DB_PORT PASS=$DB_PASSWORD && sleep 3600"]
      env:
        - name: DB_HOST           # Pod 내부에서 쓸 환경변수 이름
          valueFrom:
            configMapKeyRef:
              name: app-config    # ConfigMap 이름
              key: DB_HOST        # ConfigMap 안의 키
        - name: DB_PORT
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: DB_PORT
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: app-secret    # Secret 이름
              key: DB_PASSWORD    # Secret 안의 키
  restartPolicy: Never
```

```bash
kubectl apply -f pod-valuefrom.yaml
kubectl logs pod-valuefrom
```

```
HOST=mysql-svc PORT=3306 PASS=supersecret123
```

`envFrom`과의 차이 — 필요한 키만 선택하고, 환경변수 이름도 자유롭게 바꿀 수 있습니다.

## 5) 주입 방식 3 — volumeMount (파일로 마운트)

설정 파일 전체를 컨테이너 안의 경로에 **파일로 마운트**합니다. Nginx 설정 파일, application.properties 등을 주입할 때 사용합니다.

`pod-volume.yaml` 파일을 만듭니다.

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: pod-volume
spec:
  containers:
    - name: app
      image: busybox:1.36
      command: ["sh", "-c", "cat /config/app.properties && sleep 3600"]
      volumeMounts:
        - name: config-volume
          mountPath: /config        # 컨테이너 안의 마운트 경로
  volumes:
    - name: config-volume
      configMap:
        name: app-config
        items:
          - key: app.properties     # ConfigMap의 키
            path: app.properties    # 마운트될 파일명
  restartPolicy: Never
```

```bash
kubectl apply -f pod-volume.yaml
kubectl logs pod-volume
```

```
log.level=INFO
max.connections=100
timeout.seconds=30
```

마운트된 파일 목록 확인:

```bash
kubectl exec pod-volume -- ls /config
kubectl exec pod-volume -- cat /config/app.properties
```

## 6) ConfigMap 변경 — 반영 방식 차이

ConfigMap을 수정할 때 환경변수 방식과 볼륨 마운트 방식이 다르게 동작합니다.

```bash
kubectl edit configmap app-config
# APP_ENV: development → APP_ENV: production 으로 변경 후 저장 (:wq)
```

### 환경변수 방식 — Pod 재시작 전까지 반영 안됨

```bash
kubectl exec pod-envfrom -- env | grep APP_ENV
# APP_ENV=development  (변경 전 값 그대로)
```

환경변수는 Pod가 시작될 때 한 번만 읽습니다. 변경을 반영하려면 Pod를 재시작해야 합니다.

```bash
kubectl delete pod pod-envfrom
kubectl apply -f pod-envfrom.yaml
kubectl logs pod-envfrom | grep APP_ENV
# APP_ENV=production  (변경 후 값 반영됨)
```

### 볼륨 마운트 방식 — 약 1분 내 자동 갱신

```bash
kubectl exec pod-volume -- cat /config/app.properties
# 1분 내외로 기다렸다가 다시 실행
kubectl exec pod-volume -- cat /config/app.properties
# log.level이나 다른 값이 갱신되어 있음
```

> 볼륨 마운트 방식은 Pod 재시작 없이 설정을 핫리로드할 수 있어 운영 환경에서 유용합니다.

## 7) 존재하지 않는 ConfigMap 참조 — 에러 확인

ConfigMap 이름을 일부러 틀리게 작성해서 에러를 확인합니다.

`pod-error.yaml` 파일을 만듭니다.

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: pod-error
spec:
  containers:
    - name: app
      image: busybox:1.36
      command: ["sh", "-c", "sleep 3600"]
      envFrom:
        - configMapRef:
            name: does-not-exist    # 존재하지 않는 ConfigMap
  restartPolicy: Never
```

```bash
kubectl apply -f pod-error.yaml
kubectl get pod pod-error    # Pending 또는 Error 상태
kubectl describe pod pod-error    # Events에서 오류 메시지 확인
```

`CreateContainerConfigError: configmap "does-not-exist" not found` 메시지를 확인합니다.

```bash
kubectl delete pod pod-error
```

## 8) Namespace — 리소스의 논리적 격리 공간

지금까지 만든 ConfigMap, Secret, Pod는 모두 `default` 네임스페이스에 생성됐습니다. 네임스페이스는 클러스터 안에서 **리소스를 논리적으로 구분하는 공간**입니다.

### 현재 네임스페이스 확인

```bash
kubectl get namespaces
```

```
NAME              STATUS   AGE
default           Active   1d
kube-system       Active   1d    ← K8s 내부 컴포넌트
kube-public       Active   1d
kube-node-lease   Active   1d
```

### 새 네임스페이스 생성

개발(dev)과 운영(prod) 환경을 분리해봅니다.

```bash
kubectl create namespace dev
kubectl create namespace prod
kubectl get namespaces
```

### 네임스페이스에 ConfigMap 배포

같은 이름의 ConfigMap을 두 네임스페이스에 각각 다른 값으로 만들 수 있습니다.

```bash
# dev 환경 ConfigMap
kubectl create configmap app-config \
  --from-literal=APP_ENV=development \
  --from-literal=DB_HOST=mysql-dev \
  -n dev

# prod 환경 ConfigMap
kubectl create configmap app-config \
  --from-literal=APP_ENV=production \
  --from-literal=DB_HOST=mysql-prod \
  -n prod
```

```bash
kubectl get configmap app-config -n dev -o yaml   # APP_ENV=development
kubectl get configmap app-config -n prod -o yaml  # APP_ENV=production
```

이름은 `app-config`로 동일하지만 네임스페이스가 다르기 때문에 완전히 별개의 리소스입니다.

### ConfigMap은 같은 네임스페이스 안에서만 참조 가능

`dev` 네임스페이스의 Pod는 `dev`의 ConfigMap만 참조할 수 있습니다.

```bash
# dev 네임스페이스에 Pod 배포
kubectl run test-dev --image=busybox:1.36 \
  --restart=Never -n dev \
  --overrides='{"spec":{"containers":[{"name":"test-dev","image":"busybox:1.36","command":["sh","-c","echo $APP_ENV && sleep 3600"],"envFrom":[{"configMapRef":{"name":"app-config"}}]}],"restartPolicy":"Never"}}'

kubectl logs test-dev -n dev
# development
```

```bash
# prod 네임스페이스에 Pod 배포
kubectl run test-prod --image=busybox:1.36 \
  --restart=Never -n prod \
  --overrides='{"spec":{"containers":[{"name":"test-prod","image":"busybox:1.36","command":["sh","-c","echo $APP_ENV && sleep 3600"],"envFrom":[{"configMapRef":{"name":"app-config"}}]}],"restartPolicy":"Never"}}'

kubectl logs test-prod -n prod
# production
```

같은 `app-config`라는 이름이지만 네임스페이스별로 다른 값이 주입된 것을 확인합니다.

### YAML에 네임스페이스 명시

`metadata.namespace`에 네임스페이스를 지정하면 `-n` 없이도 원하는 네임스페이스에 배포됩니다.

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
  namespace: dev        # 네임스페이스 명시
data:
  APP_ENV: development
```

### 네임스페이스별 리소스 조회

```bash
kubectl get all -n dev     # dev 네임스페이스 전체 리소스
kubectl get all -n prod    # prod 네임스페이스 전체 리소스
kubectl get configmaps -A  # 모든 네임스페이스의 ConfigMap
```

### 정리

```bash
kubectl delete namespace dev
kubectl delete namespace prod
```

> 네임스페이스를 삭제하면 그 안의 모든 리소스(Pod, ConfigMap, Secret 등)가 함께 삭제됩니다.

## 정리 (리소스 삭제)

```bash
kubectl delete pod pod-envfrom pod-valuefrom pod-volume
kubectl delete configmap app-config
kubectl delete secret app-secret
```

## 주입 방식 비교

| 방식 | 사용 상황 | 환경변수 이름 변경 | 동적 갱신 |
|---|---|---|---|
| `envFrom` | 전체 키를 한번에 환경변수로 | 불가 | Pod 재시작 필요 |
| `env.valueFrom` | 특정 키만 골라서 환경변수로 | 가능 | Pod 재시작 필요 |
| `volumeMount` | 설정 파일 형태로 마운트 | — | 자동 갱신 (약 1분) |

## ConfigMap vs Secret 비교

| 항목 | ConfigMap | Secret |
|---|---|---|
| 용도 | 비민감 설정 (DB 주소, 포트, 환경 이름) | 민감 정보 (비밀번호, API 키, 인증서) |
| 저장 방식 | 평문 | base64 인코딩 |
| `kubectl describe` | 값이 그대로 보임 | 값이 숨겨짐 |
| Git 관리 | 가능 | 직접 커밋 비권장 |

## 트러블슈팅

| 증상 | 확인 사항 |
|---|---|
| Pod `CreateContainerConfigError` | ConfigMap·Secret 이름 오타 또는 아직 생성 전 |
| 환경변수 값이 비어있음 | `key` 이름이 ConfigMap의 실제 키와 일치하는지 확인 |
| base64 디코딩 오류 | `echo -n` 옵션 누락 여부 확인 |
| ConfigMap 변경이 반영 안됨 | 환경변수 방식은 Pod 재시작 필요, 볼륨 방식은 1분 대기 |
