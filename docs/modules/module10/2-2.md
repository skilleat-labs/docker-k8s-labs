# 2-2. Service 실습 (ClusterIP · NodePort · port-forward)

## 실습 목표

- ClusterIP, NodePort Service를 생성하고 각 타입의 차이를 체감한다.
- 클러스터 내 모든 Pod에서 ClusterIP로 통신이 가능함을 직접 확인한다.
- Pod 내부에서 Service 이름(DNS), ClusterIP, NodePort 세 가지 방식으로 모두 통신해본다.
- label selector 기반 라우팅 동작을 확인한다.

## 전제 조건

2-1 실습에서 만든 `rollout-deploy`가 실행 중이어야 합니다.

```bash
kubectl get pods -l app=rollout
```

없으면 재생성합니다.

```bash
kubectl apply -f deploy-rollout.yaml
```

---

## Service란?

Pod는 재시작할 때마다 IP가 바뀝니다. Service는 **고정된 진입점**(ClusterIP 또는 NodePort)을 제공하여 Pod IP가 바뀌어도 항상 같은 주소로 접근할 수 있게 합니다.

```
클러스터 내부 어느 Pod에서든
  → Service 이름(DNS) 또는 ClusterIP 로 접근
    → Service가 label selector로 대상 Pod를 찾아 트래픽 전달
```

---

## 1) ClusterIP Service 생성

`svc-clusterip.yaml` 파일을 만듭니다.

```yaml
apiVersion: v1
kind: Service
metadata:
  name: rollout-svc
spec:
  type: ClusterIP
  selector:
    app: rollout
  ports:
    - port: 80
      targetPort: 80
```

```bash
kubectl apply -f svc-clusterip.yaml
kubectl get svc rollout-svc
```

출력 예시:

```
NAME          TYPE        CLUSTER-IP     EXTERNAL-IP   PORT(S)   AGE
rollout-svc   ClusterIP   10.96.45.123   <none>        80/TCP    10s
```

- `CLUSTER-IP`: 클러스터 내부에서만 유효한 가상 IP. 이 IP는 Pod가 재시작되어도 변하지 않습니다.
- `EXTERNAL-IP`: `<none>` — 클러스터 외부에서 직접 접근 불가

### Endpoints 확인

Service가 실제로 어떤 Pod에 연결되어 있는지 확인합니다.

```bash
kubectl get endpoints rollout-svc
kubectl get pods -o wide    # Endpoints IP와 Pod IP 비교
```

Pod IP 목록과 Endpoints IP 목록이 일치해야 합니다.

---

## 2) Pod 내부에서 ClusterIP로 통신

클러스터에 올라간 **모든 Pod**는 다른 Pod의 Service에 ClusterIP로 접근할 수 있습니다.

### 방법 A — ClusterIP 직접 사용

`rollout-svc`의 CLUSTER-IP를 확인합니다.

```bash
kubectl get svc rollout-svc    # CLUSTER-IP 값 메모 (예: 10.96.45.123)
```

임시 curl Pod를 실행해 ClusterIP로 요청을 보냅니다.

```bash
kubectl run curl-test --image=curlimages/curl:latest --restart=Never -it --rm \
  -- curl http://10.96.45.123
```

파란색(v1.0.0) 응답이 오면 성공입니다.

> 이 curl Pod는 `rollout-deploy`와 **전혀 다른 Pod**이지만, 같은 클러스터 안에 있기 때문에 ClusterIP로 통신이 됩니다. 클러스터 내 모든 Pod는 서로의 Service에 접근할 수 있습니다.

### 방법 B — Service 이름(DNS)으로 접근

IP를 외울 필요 없이 Service 이름을 DNS로 사용할 수 있습니다.

```bash
kubectl run curl-test --image=curlimages/curl:latest --restart=Never -it --rm \
  -- curl http://rollout-svc
```

K8s 내부 DNS(CoreDNS)가 `rollout-svc`를 자동으로 ClusterIP로 변환해줍니다.

FQDN(전체 도메인) 형식으로도 접근됩니다.

```bash
kubectl run curl-test --image=curlimages/curl:latest --restart=Never -it --rm \
  -- curl http://rollout-svc.default.svc.cluster.local
```

DNS 이름 형식: `<service이름>.<namespace>.svc.cluster.local`

### 방법 C — 실행 중인 Pod 내부에서 접근

이미 실행 중인 Pod에 접속해서 Service로 통신합니다.

```bash
# rollout-deploy의 Pod 중 하나에 접속
kubectl exec -it deployment/rollout-deploy -- sh

# Pod 내부에서 실행
curl http://rollout-svc         # Service 이름으로 접근
curl http://10.96.45.123        # ClusterIP로 접근 (위에서 메모한 IP)
exit
```

> **포인트**: 같은 Deployment의 Pod끼리는 물론, 전혀 다른 Deployment의 Pod에서도 Service 이름 또는 ClusterIP로 접근할 수 있습니다.

## 3) label selector 동작 확인

Service는 label selector로 트래픽을 보낼 Pod를 결정합니다.

```bash
kubectl describe svc rollout-svc
```

확인 항목:

- `Selector`: `app=rollout` — 이 label을 가진 Pod에만 트래픽 전달
- `Endpoints`: 현재 연결된 Pod IP 목록
- `Port` / `TargetPort`: Service 포트 → Pod 포트 매핑

### label이 다른 Pod는 제외됨을 확인

```bash
kubectl run other-pod --image=nginx:1.25 --labels="app=other"
kubectl get endpoints rollout-svc    # other-pod IP가 포함되지 않음
kubectl delete pod other-pod
```

## 4) NodePort Service 생성

NodePort는 ClusterIP 기능에 더해 **노드의 IP:포트**로도 외부에서 접근할 수 있게 합니다.

`svc-nodeport.yaml` 파일을 만듭니다.

```yaml
apiVersion: v1
kind: Service
metadata:
  name: rollout-nodeport
spec:
  type: NodePort
  selector:
    app: rollout
  ports:
    - port: 80
      targetPort: 80
      nodePort: 30080
```

```bash
kubectl apply -f svc-nodeport.yaml
kubectl get svc rollout-nodeport
```

출력 예시:

```
NAME               TYPE       CLUSTER-IP      EXTERNAL-IP   PORT(S)        AGE
rollout-nodeport   NodePort   10.96.78.200    <none>        80:30080/TCP   10s
```

`80:30080/TCP` — Service 내부 포트(80)와 노드 포트(30080)가 함께 표시됩니다.

### 노드 IP 확인

```bash
kubectl get nodes -o wide    # INTERNAL-IP 확인
```

### 세 가지 방법으로 모두 접근

```bash
# 1. 로컬에서 localhost:nodePort (Docker Desktop/Rancher Desktop)
curl http://localhost:30080

# 2. 노드 IP:nodePort
curl http://<노드 INTERNAL-IP>:30080

# 3. ClusterIP (클러스터 내부 Pod에서)
kubectl run curl-test --image=curlimages/curl:latest --restart=Never -it --rm \
  -- curl http://rollout-nodeport
```

> NodePort Service도 ClusterIP를 함께 가집니다. 클러스터 내부에서는 Service 이름이나 ClusterIP로, 외부에서는 노드IP:NodePort로 접근합니다.

---

## 5) port-forward (개발 시 빠른 접속)

`port-forward`는 Service나 Pod에 로컬 포트를 임시로 연결하는 방법입니다. 클러스터 외부 접근 없이 개발·디버깅 시 사용합니다.

```bash
kubectl port-forward svc/rollout-svc 8080:80
```

새 터미널에서:

```bash
curl http://localhost:8080
```

`Ctrl+C`로 종료.

---

## 6) Pod 삭제 후 Service 연속성 확인

Pod가 재시작돼도 Service를 통한 접근이 끊기지 않는지 확인합니다.

터미널 1 — 반복 요청:

=== "macOS / Linux"
    ```bash
    while true; do curl -s -o /dev/null -w "%{http_code}\n" http://localhost:30080; sleep 1; done
    ```

=== "Windows (PowerShell)"
    ```powershell
    while ($true) { (curl.exe -s -o NUL -w "%{http_code}" http://localhost:30080); Start-Sleep 1 }
    ```

터미널 2 — Pod 강제 삭제:

```bash
kubectl delete pod -l app=rollout --wait=false
kubectl get pods -w    # 새 Pod 생성 확인
```

터미널 1의 응답 코드가 계속 `200`으로 유지되면 Service가 새 Pod로 트래픽을 자동 전환한 것입니다.

`Ctrl+C`로 종료.

## Service 타입 비교

| 타입 | 접근 방법 | 접근 범위 | 주요 용도 |
|---|---|---|---|
| ClusterIP | Service명(DNS) 또는 ClusterIP | 클러스터 내부 Pod 전체 | 서비스 간 내부 통신 |
| NodePort | 노드IP:NodePort | 클러스터 외부 | 개발·테스트 환경 외부 노출 |
| LoadBalancer | 클라우드 LB IP | 클러스터 외부 | 프로덕션 외부 노출 (AKS 등) |

## 트러블슈팅

| 증상 | 확인 사항 |
|---|---|
| Endpoints가 `<none>` | Pod label과 Service selector 일치 여부 확인 |
| ClusterIP로 접근 불가 | Pod 내부에서 실행했는지 확인 (로컬 터미널에서는 불가) |
| NodePort 접속 불가 | Docker Desktop에서 `localhost:30080` 사용, 노드 IP는 `kubectl get nodes -o wide` 확인 |
| DNS 이름으로 접근 불가 | Service와 Pod가 같은 네임스페이스에 있는지 확인 |
