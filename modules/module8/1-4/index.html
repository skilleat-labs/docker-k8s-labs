    1-4. 컨테이너 볼륨 (Volume) 실습¶

    실습 목표¶

    • 컨테이너의 데이터 유실(Ephemeral) 문제 직접 체험하기
    • Docker Volume, Bind Mount, tmpfs의 차이점 이해하기
    • 각 볼륨 타입을 상황에 맞게 선택하여 사용하기

    예상 소요 시간 (약 30분)¶

    • 0~5분: 데이터 유실 문제 확인
    • 5~15분: Docker Volume 생성 및 실습
    • 15~23분: Bind Mount 실습
    • 23~30분: 정리 및 볼륨 제거

    사전 조건¶

    • Docker 설치 완료
    • 기본 터미널 명령어 이해

    확인:

    docker --version
docker ps

    1) 컨테이너 데이터 유실 문제 체험¶

    1-1. 컨테이너 내 파일 생성 및 확인¶

    docker run -it --name temp-test skilleatlab.azurecr.io/lab/busybox:latest

    컨테이너 내부에서:

    # 파일 생성
echo "This is important data" > /data.txt
cat /data.txt

# 디렉토리도 생성
mkdir -p /myapp/config
echo "config value=123" > /myapp/config/settings.txt

# 현재 상태 확인
ls -la /
ls -la /myapp/config

exit

    1-2. 컨테이너 삭제 후 데이터 확인¶

    # 컨테이너 삭제
docker rm temp-test

# 같은 이름으로 새 컨테이너 생성
docker run -it --name temp-test skilleatlab.azurecr.io/lab/busybox:latest

    컨테이너 내부에서:

    # 이전에 만든 파일이 있는지 확인
cat /data.txt  # 오류 발생
ls -la /myapp  # 디렉토리가 없음

exit

    관찰 포인트:

    • 새 컨테이너는 이미지의 깨끗한 상태로 시작
    • 이전 컨테이너의 모든 변경사항이 사라짐
    • 이것이 바로 "Ephemeral" 특성
    docker rm temp-test

    2) Docker Volume으로 데이터 보존하기¶

    2-1. 볼륨 생성 및 확인¶

    # 볼륨 생성
docker volume create mydata

# 생성된 볼륨 확인
docker volume ls

# 상세 정보 확인
docker volume inspect mydata

    관찰 사항:

    • Mountpoint: 호스트에서 실제 저장되는 위치
    • 도커가 자동으로 관리하는 저장소

    2-2. 볼륨을 마운트한 컨테이너 실행¶

    docker run -it --name vol-test -v mydata:/data skilleatlab.azurecr.io/lab/busybox:latest

    컨테이너 내부에서:

    # /data 디렉토리에 여러 파일 생성
echo "Volume data 1" > /data/file1.txt
echo "Volume data 2" > /data/file2.txt
mkdir /data/subdir
echo "Nested file" > /data/subdir/file3.txt

# 현재 상태 확인
ls -la /data
ls -la /data/subdir

exit

    2-3. 컨테이너 삭제 후 같은 볼륨으로 새 컨테이너 생성¶

    # 첫 번째 컨테이너 삭제
docker rm vol-test

# 같은 볼륨으로 새 컨테이너 실행
docker run -it --name vol-test2 -v mydata:/data skilleatlab.azurecr.io/lab/busybox:latest

    컨테이너 내부에서:

    # 이전에 만든 파일이 여전히 존재!
ls -la /data
cat /data/file1.txt
cat /data/subdir/file3.txt

# 새 파일도 추가 가능
echo "New data from container 2" >> /data/file1.txt
cat /data/file1.txt

exit

    2-4. 여러 컨테이너가 같은 볼륨 공유하기¶

    vol-test2가 아직 실행 중인 상태에서 진행합니다. 새 터미널을 열고 VM에 SSH 접속한 뒤 아래 명령어를 실행하세요.

    docker run -it --name vol-test3 -v mydata:/data skilleatlab.azurecr.io/lab/busybox:latest

    컨테이너 내부에서:

    # 다른 컨테이너가 작성한 파일도 볼 수 있음
cat /data/file1.txt  # 컨테이너 2에서 수정된 내용 포함

# 이 컨테이너에서도 수정
echo "Updated by container 3" > /data/file1.txt

exit

    정리:

    docker rm vol-test2 vol-test3
docker volume rm mydata

    핵심 학습:

    • Volume은 컨테이너 삭제 후에도 데이터 유지
    • 여러 컨테이너가 같은 볼륨을 공유 가능
    • 도커가 자동으로 관리하므로 편리함

    3) Bind Mount 실습¶

    3-1. 호스트 디렉토리 준비¶

    # 호스트에서 실행
mkdir -p ~/bind-test/config
mkdir -p ~/bind-test/logs

# 초기 파일 생성
echo "host_config=production" > ~/bind-test/config/app.conf
echo "2026-04-28 App started" > ~/bind-test/logs/app.log

# 호스트에서 확인
ls -la ~/bind-test
cat ~/bind-test/config/app.conf

    3-2. Bind Mount로 컨테이너 실행¶

    docker run -it --name bind-test -v ~/bind-test:/app skilleatlab.azurecr.io/lab/busybox:latest

    컨테이너 내부에서:

    # 호스트의 파일이 보임
ls -la /app
cat /app/config/app.conf
cat /app/logs/app.log

# 호스트의 파일을 컨테이너에서 수정
echo "Updated by container" >> /app/config/app.conf

# 새로운 파일 생성
echo "2024-01-17 Processing complete" >> /app/logs/app.log
echo "process_status=completed" > /app/status.txt

exit

    3-3. 호스트에서 변경사항 확인¶

    # 컨테이너에서 수정한 파일이 호스트에 반영됨
cat ~/bind-test/config/app.conf
cat ~/bind-test/logs/app.log
cat ~/bind-test/status.txt

# 호스트에서 새 파일 생성
echo "host_added_data=yes" > ~/bind-test/new_from_host.txt

    3-4. 같은 Bind Mount로 새 컨테이너 실행¶

    docker run -it --name bind-test2 -v ~/bind-test:/app skilleatlab.azurecr.io/lab/busybox:latest

    컨테이너 내부에서:

    # 호스트에서 추가한 파일도 보임 (실시간 동기화)
ls -la /app
cat /app/new_from_host.txt

exit

    정리:

    docker rm bind-test bind-test2

    핵심 학습:

    • Bind Mount: 호스트 경로를 직접 컨테이너에 연결
    • 양방향 실시간 동기화
    • 개발 환경에서 코드 수정 시 유용 (핫 리로드)

    4) 정리 및 상황별 선택 기준¶

    5-1. 모든 컨테이너 및 볼륨 정리¶

    # 실행 중인 모든 컨테이너 중지
docker stop $(docker ps -q) 2>/dev/null

# 모든 컨테이너 삭제
docker rm $(docker ps -aq) 2>/dev/null

# 사용 중이지 않은 볼륨 삭제
docker volume prune -f

# 정리 확인
docker ps -a
docker volume ls

    5-2. 상황별 선택 기준¶

    상황 추천 방식 이유
    데이터 영구 보존 Volume 도커 관리, 백업 용이
    개발 중 코드 수정 Bind Mount 실시간 동기화, 핫 리로드
    여러 컨테이너 공유 Volume 안정적, 접근 제어
    호스트 특정 경로 사용 Bind Mount 호스트 경로 직접 연결

    📝 핵심 요약¶

    1. Ephemeral 특성: 컨테이너 삭제 시 기본적으로 모든 데이터 손실
    2. 2가지 볼륨:
    3. Volume: 도커 관리, 프로덕션 권장
    4. Bind Mount: 호스트 경로, 개발 권장
    5. 선택 기준: 데이터 보존 필요성과 사용 환경에 따라 결정
