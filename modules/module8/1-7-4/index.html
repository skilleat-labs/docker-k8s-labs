    1-7-4. 마이크로서비스 아키텍처 체험¶

    실습 목표¶

    • 1-7-3에서 체험한 모놀리식의 한계를 마이크로서비스 구조로 어떻게 해결하는지 확인하기
    • 서비스별 독립 배포와 장애 격리를 직접 체험하기
    • 스케일 아웃 시 데이터 일관성이 유지되는 것 확인하기
    • 모놀리식과 마이크로서비스를 나란히 비교하기

    예상 소요 시간 (약 40분)¶

    • 0~10분: MSA 앱 실행 및 구조 확인
    • 10~20분: 장애 격리 체험
    • 20~30분: 독립 배포 체험
    • 30~40분: 스케일 아웃 + 데이터 일관성 확인

    사용 이미지¶

    서비스 이미지 역할
    frontend skilleatlab.azurecr.io/msa-cafe-frontend:v1 nginx — 정적 파일 서빙 + API 프록시
    backend skilleatlab.azurecr.io/msa-cafe-backend:v1 Flask — 상품·주문 API
    db mysql:8.0 공유 데이터베이스 
    브라우저
   ↓
[frontend: nginx]  ← 정적 파일 서빙
   ↓ /api/* 프록시
[backend: Flask]   ← API 처리
   ↓
[db: MySQL]        ← 데이터 영속성

    사전 정리¶

    docker rm -f $(docker ps -aq) 2>/dev/null || true
docker ps -a

    1) MSA 앱 실행¶

    1-1. 실습 디렉토리 생성¶

    mkdir ~/msa-cafe-lab && cd ~/msa-cafe-lab

    1-2. docker-compose.yml 작성¶

    nano docker-compose.yml

    저장: Ctrl+O → Enter → 종료: Ctrl+X

    services:
  frontend:
    image: skilleatlab.azurecr.io/msa-cafe-frontend:v1
    ports:
      - "5010:80"
    depends_on:
      - backend

  backend:
    image: skilleatlab.azurecr.io/msa-cafe-backend:v1
    environment:
      DB_HOST: db
      DB_USER: cafe
      DB_PASS: cafe1234
      DB_NAME: cafedb
    depends_on:
      - db

  db:
    image: mysql:8.0
    environment:
      MYSQL_DATABASE: cafedb
      MYSQL_USER: cafe
      MYSQL_PASSWORD: cafe1234
      MYSQL_ROOT_PASSWORD: rootpass
    volumes:
      - cafe-data:/var/lib/mysql

volumes:
  cafe-data:

    1-3. 실행¶

    docker compose up -d

    MySQL이 완전히 뜰 때까지 backend가 재시도합니다. 30초~1분 후 브라우저에서 확인합니다.

    http://192.168.56.10:5010

    1-4. 아키텍처 패널 읽기¶

    화면 하단의 ✅ 마이크로서비스 아키텍처 정보 패널을 확인합니다.

    관찰 포인트:

    • Frontend hostname — nginx 컨테이너 식별자
    • Backend hostname — Flask 컨테이너 식별자
    • 두 hostname이 서로 다릅니다 — 실제로 다른 컨테이너에서 실행 중
    • Database — MySQL, 컨테이너 외부 별도 서비스

    지금 보이는 Frontend · Backend hostname을 기억해두세요.

    1-5. 주문 몇 개 넣어두기¶

    메뉴에서 3~4개 주문합니다. 주문 내역이 쌓이는 것을 확인합니다.

    2) 장애 격리 체험¶

    1-7-3에서는 docker pause로 전체 서비스가 중단됐습니다. MSA에서는 어떻게 다른지 확인합니다.

    2-1. backend만 일시 정지¶

    docker compose pause backend

    브라우저를 새로고침합니다.

    http://192.168.56.10:5010

    관찰 포인트:

    • 상품 카드 목록 → 로딩 실패 (backend가 없으므로)
    • 주문 내역 → 로딩 실패 (backend가 없으므로)
    • 페이지 자체(HTML) → 정상 응답 (frontend는 살아있으므로)

    브라우저 개발자 도구 → Network 탭에서 /api/products 요청은 실패하지만 HTML 자체는 200으로 뜨는 것을 확인할 수 있습니다.

    2-2. 복구¶

    docker compose unpause backend

    새로고침하면 정상 동작합니다.

    모놀리식과 비교

    1-7-3에서는 docker pause 하나로 모든 것이 멈췄습니다. MSA에서는 backend가 멈춰도 frontend 컨테이너는 계속 응답합니다. 장애가 해당 서비스에만 격리됩니다.

    3) 독립 배포 체험¶

    backend만 재배포해도 frontend는 계속 서비스됩니다.

    3-1. 주문 내역 확인¶

    현재 주문 내역이 있는 상태를 확인합니다.

    http://192.168.56.10:5010

    3-2. backend만 재시작¶

    docker compose restart backend

    브라우저를 새로고침합니다.

    관찰 포인트:

    • 페이지 → 즉시 응답 (frontend는 재시작하지 않았으므로)
    • 주문 내역 → backend 재시작 후 잠시 후 다시 표시됨
    • 데이터 → MySQL에 저장되어 있어 주문 내역이 유지됩니다

    모놀리식과 비교

    1-7-3에서는 재배포 시 두 가지 문제가 동시에 발생했습니다.

    모놀리식 MSA
    재배포 중 페이지 전체 중단 frontend 정상 응답
    재배포 후 데이터 사라짐 (SQLite 초기화) 유지됨 (MySQL 보존)

    3-3. backend hostname 확인¶

    http://192.168.56.10:5010

    아키텍처 패널에서 backend hostname이 바뀐 것을 확인합니다. 새 컨테이너로 교체됐지만 데이터는 그대로입니다.

    4) 스케일 아웃 + 데이터 일관성¶

    4-1. backend를 3개로 늘리기¶

    docker compose up -d --scale backend=3

    docker compose ps

    backend 컨테이너가 3개 실행 중인 것을 확인합니다.

    4-2. 주문하면서 hostname 변화 관찰¶

    브라우저에서 주문을 여러 번 합니다.

    http://192.168.56.10:5010

    관찰 포인트:

    • 아키텍처 패널의 Backend hostname이 요청마다 바뀝니다 (nginx가 3개 backend 중 하나로 라우팅)
    • 주문 내역은 모든 backend에서 동일하게 보입니다 (MySQL을 공유하기 때문)

    모놀리식과 비교

    1-7-3에서 2개 인스턴스를 띄웠을 때 각자 다른 주문 데이터를 가졌습니다. MSA에서는 backend가 몇 개든 MySQL을 공유하므로 데이터가 일관됩니다.

    4-3. frontend는 그대로¶

    docker compose ps

    frontend는 여전히 1개입니다. 트래픽이 많은 backend만 선택적으로 늘릴 수 있습니다.

    4-4. 정리¶

    docker compose down -v

    5) 모놀리식 vs 마이크로서비스 최종 비교¶

    1-7-3과 1-7-4에서 직접 체험한 내용을 정리합니다.

    상황 모놀리식 (1-7-3) 마이크로서비스 (1-7-4)
    장애 발생 시 전체 서비스 중단 해당 서비스만 영향
    배포 중 페이지 전체 중단 frontend는 계속 응답
    배포 후 데이터 SQLite 초기화 → 사라짐 MySQL 보존 → 유지
    스케일 아웃 전체 복제, 데이터 불일치 필요한 서비스만, 데이터 일관
    서비스 독립성 없음 (하나의 배포 단위) 있음 (서비스별 독립 배포)

    그래서 Kubernetes가 필요합니다

    MSA 구조는 모놀리식의 한계를 해결하지만 새로운 과제가 생깁니다.

    • 서비스가 늘어날수록 어떤 컨테이너가 어디서 실행되는지 추적이 어려워집니다
    • 컨테이너가 죽었을 때 자동으로 복구되지 않습니다
    • 트래픽에 따라 자동으로 스케일되지 않습니다
    • 수십 개의 서비스를 docker compose scale 로 관리하는 건 한계가 있습니다

    이것이 Kubernetes(Module 10) 로 넘어가는 이유입니다.

    📝 핵심 요약¶

    1. 장애 격리 — MSA에서는 backend가 멈춰도 frontend는 계속 응답
    2. 독립 배포 — backend만 재시작해도 frontend는 중단 없이 서비스
    3. 데이터 영속성 — MySQL이 별도 서비스이므로 backend 재시작과 무관
    4. 선택적 스케일 — 부하가 많은 서비스만 늘릴 수 있음
    5. 다음 단계 — 서비스가 늘어날수록 오케스트레이션 도구(K8s)가 필요해짐

    📄 평가 대비 — CA 전략 보고서 작성 가이드¶

    1-7-3과 1-7-4에서 체험한 내용이 보고서의 핵심입니다. 아래 구조로 A4 1~2페이지 분량을 작성하면 됩니다.

    1. 모놀리식 구조의 한계 (12점 — 2가지 이상 서술)¶

    1-7-3에서 직접 체험한 문제를 시나리오(ShopCore)에 맞게 서술합니다.

    문제 1-7-3 체험 보고서 서술 예시
    장애 전파 docker pause → 전체 중단 "결제 기능 과부하 시 상품 조회·주문 기능도 함께 중단됨"
    배포 중단 재배포 시 서비스 중단 + 데이터 유실 "알림 기능 수정 배포 시에도 전체 서비스 재시작 필요, 중단 발생"
    스케일아웃 불가 전체 복제 → 데이터 불일치 "블랙프라이데이 결제 폭주 시 결제만 스케일 불가, 전체 복제해야 함"

    2. 서비스 분리 설계 (10점)¶

    3개 서비스의 분리 이유와 팀 매핑 표를 포함합니다.

    서비스 분리 이유:

    서비스 역할 분리 이유
    Order Service 주문·결제 처리 결제는 장애 민감도가 높아 독립 배포·스케일 필요
    Product Service 상품·재고 관리 상품 데이터 변경 빈도가 낮아 독립 운영 효율적
    Notification Service 알림 발송 비동기 처리 가능, 다른 서비스 장애와 무관하게 운영

    팀 ↔ 서비스 매핑:

    팀 담당 서비스 독립 배포 가능
    결제팀 Order Service ✅
    상품팀 Product Service ✅
    플랫폼팀 Notification Service ✅

    3. MSA 도입 권고 (5점 — 이유 1가지 이상)¶

    도입 권고 여부를 명확히 쓰고, 근거를 1가지 이상 제시합니다.

    예시:

    ShopCore에 MSA 전환을 권고합니다.

    기술적 관점: 결제 기능만 독립 스케일 아웃이 가능해지므로 블랙프라이데이와 같은 트래픽 집중 상황에서 전체 서비스 중단 없이 대응할 수 있습니다.

    심화: Conway's Law (4점 — 선택)¶

    쓰면 추가 4점이지만 안 써도 70점 달성에 문제 없습니다.

    "조직이 설계하는 시스템 구조는 그 조직의 커뮤니케이션 구조를 닮게 된다" — Melvin Conway

    ShopCore에 적용 예시: 결제팀·상품팀·플랫폼팀이 독립된 조직이라면, 각 팀이 담당하는 서비스를 독립 배포할 수 있는 MSA 구조가 자연스럽게 적합합니다.
