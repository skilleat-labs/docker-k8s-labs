    1-7. Docker Compose 기본 실습¶

    실습 목표¶

    • docker-compose.yml 파일 구조 이해하기
    • services, networks, volumes의 역할 익히기
    • Compose로 여러 컨테이너를 한 번에 실행/중지/정리하기

    예상 소요 시간 (약 30분)¶

    • 0~5분: Compose 개념과 파일 구조 이해
    • 5~15분: 간단한 웹 + DB Compose 파일 작성
    • 15~25분: Compose 명령어로 실행, 로그 확인, 서비스 연결 확인
    • 25~30분: 정리 및 추가 실습

    사전 조건¶

    • Docker 설치 완료
    • Docker Compose가 설치되었거나 Docker Desktop 사용 중
    • 기본 Docker 명령어 실행 경험

    확인:

    docker --version
docker compose version

    1) Docker Compose 개념 이해¶

    Docker Compose는 여러 컨테이너를 하나의 YAML 파일로 정의하고, 한 번에 실행/제어하도록 돕는 도구입니다.

    docker-compose.yml의 핵심 최상위 키:

    • services: 실행할 컨테이너 목록
    • networks: 서비스 간 네트워크 연결 정의
    • volumes: 데이터를 유지할 볼륨 정의

    참고: 최신 Docker Compose(v2)에서는 version: 키를 생략해도 됩니다.

    2) 실습 디렉토리 생성¶

    Compose 파일은 반드시 별도 폴더에서 시작합니다.

    mkdir ~/compose-lab && cd ~/compose-lab

    폴더 이름 = Compose 프로젝트 이름

    Docker Compose는 docker-compose.yml이 있는 폴더 이름을 프로젝트 이름으로 자동 사용합니다.

    프로젝트 이름은 컨테이너, 네트워크, 볼륨 이름의 앞에 prefix로 붙습니다.

    예를 들어 폴더 이름이 compose-lab이면:

    리소스 실제 생성되는 이름
    web 컨테이너 compose-lab-web-1
    db 컨테이너 compose-lab-db-1
    frontend 네트워크 compose-lab_frontend
    db-data 볼륨 compose-lab_db-data

    이 덕분에 같은 Compose 파일을 다른 폴더에서 실행해도 충돌 없이 독립적으로 동작합니다.

    3) 간단한 Compose 파일 작성¶

    ~/compose-lab 디렉토리 안에서 docker-compose.yml 파일을 작성합니다.

    nano docker-compose.yml

    아래 내용을 직접 타이핑하면서 YAML 문법을 익혀보세요. 들여쓰기는 반드시 스페이스 2칸을 사용합니다 (탭 사용 불가).

    services:
  web:
    image: nginx:alpine
    ports:
      - "8888:80"
    networks:
      - frontend
    depends_on:
      - db

  db:
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: example
    volumes:
      - db-data:/var/lib/mysql
    networks:
      - backend

networks:
  frontend:
  backend:

volumes:
  db-data:

    nano 저장 방법

    내용을 붙여넣은 후:

    1. Ctrl + O → 저장 (파일명 확인 후 Enter)
    2. Ctrl + X → nano 종료

    설명¶

    • web 서비스: Nginx 웹 서버를 8888 포트로 노출
    • db 서비스: MySQL을 db-data 볼륨과 함께 실행
    • networks: frontend, backend 2개 네트워크를 정의
    • volumes: db-data 볼륨을 생성해 DB 데이터를 보존
    • depends_on: web이 시작되기 전에 db가 먼저 시작되도록 순서를 지정

    depends_on이란?

    depends_on은 컨테이너의 시작 순서를 지정합니다.

    web → depends_on: db

    위 설정은 "db 컨테이너가 먼저 시작된 후에 web을 시작하라"는 의미입니다.

    DB가 준비되기 전에 웹 서버가 먼저 뜨면 연결 오류가 발생할 수 있기 때문에, 의존 관계가 있는 서비스는 항상 순서를 지정해주는 것이 좋습니다.

    단, depends_on은 컨테이너가 시작됐는지만 보장합니다. DB가 완전히 준비(ready)됐는지는 보장하지 않습니다. 완전한 헬스체크가 필요하면 condition: service_healthy를 추가로 사용합니다.

    YAML 콜론 주의

    최상위 volumes:, networks: 아래 항목들은 값 없이 콜론만 있습니다.

    volumes:
  db-data:      # 값이 없어도 유효한 YAML — "값 없는 mapping"으로 처리됨

    이는 YAML 문법상 db-data: null과 동일하며, Docker Compose는 이를 "기본 옵션으로 볼륨을 생성하라"로 해석합니다. 콜론 뒤에 실수로 -(리스트 형식)를 쓰면 volumes must be a mapping 오류가 발생하니 주의하세요.

    4) Compose 실행¶

    docker compose up -d

    실행 후 확인:

    docker compose ps
docker compose logs --tail=20

    웹 브라우저에서 VM IP로 접속하면 Nginx 기본 페이지가 보입니다.

    http://192.168.56.10:8888

    5) 서비스 연결 확인¶

    Compose는 같은 네트워크에 있는 서비스 이름을 DNS로 사용합니다. web에서 db로 ping을 시도해봅니다.

    docker compose exec web ping -c 3 db

    실패할 것입니다. 잠깐 생각해보세요.

    현재 compose 파일에서 web과 db는 어떤 네트워크에 각각 속해 있나요? 두 서비스가 공유하는 네트워크가 있나요?

    이유 확인

    web은 frontend, db는 backend 네트워크에만 속해 있습니다. 두 서비스 사이에 공유 네트워크가 없기 때문에 DNS 해석이 되지 않아 bad address 'db' 오류가 발생합니다.

    네트워크 추가 후 재테스트¶

    docker-compose.yml에서 db 서비스에 frontend 네트워크를 추가합니다.

      db:
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: example
    volumes:
      - db-data:/var/lib/mysql
    networks:
      - backend
      - frontend   # 추가

    Compose를 재시작합니다.

    docker compose down
docker compose up -d

    다시 ping을 시도합니다.

    docker compose exec web ping -c 3 db

    이번에는 성공합니다. web과 db가 frontend 네트워크를 공유하기 때문입니다.

    6) 네트워크와 볼륨 확인¶

    docker network ls
docker volume ls

    Compose로 만든 네트워크와 볼륨 이름은 일반적으로 현재 디렉터리 이름을 접두사로 사용합니다.

    docker network inspect $(docker network ls --filter name=frontend -q)

    6-1) 네트워크 격리 — internal: true¶

    외부에 노출되는 서비스와 내부 서비스를 네트워크로 분리할 수 있습니다. internal: true 를 적용하면 해당 네트워크는 인터넷 연결이 차단됩니다.

    nano docker-compose.yml

    저장: Ctrl+O → Enter → 종료: Ctrl+X

    services:
  web:
    image: nginx:alpine
    ports:
      - "8888:80"
    networks:
      - frontend

  api:
    image: skilleatlab.azurecr.io/env-lab:v1
    networks:
      - frontend
      - backend

  db:
    image: redis:alpine
    networks:
      - backend        # 외부에 노출되지 않음

networks:
  frontend:            # 외부 접근 허용
  backend:
    internal: true     # 인터넷 차단 — 컨테이너끼리만 통신 가능

    docker compose up -d
docker network ls

    관찰 포인트:

    • web → frontend 네트워크만 → 외부 포트 바인딩 가능
    • api → 두 네트워크 모두 → web과도, db와도 통신 가능
    • db → backend 네트워크만 → 외부에서 직접 접근 불가

    internal: true 네트워크 내 컨테이너에서 인터넷이 차단되는지 확인합니다.

    docker compose exec db ping -c 2 8.8.8.8

    Network is unreachable 또는 타임아웃 → internal: true 적용 확인.

    docker compose down

    7) 정리¶

    Compose 실행을 멈추고 리소스를 정리합니다.

    docker compose down

    데이터 볼륨까지 제거하려면:

    docker compose down -v

    8) 도전 과제 : 미니 블로그를 Compose로¶

    시작 전에 현재 실행 중인 컨테이너를 모두 정리합니다.

    # 실행 중인 컨테이너 전체 중지 및 삭제
docker rm -f $(docker ps -aq) 2>/dev/null || true

# 확인
docker ps -a

    1-6-2 도전 과제에서 docker run으로 직접 실행했던 미니 블로그를 이번에는 docker-compose.yml 하나로 구성해보세요.

    조건

    항목 내용
    디렉토리 새 디렉토리를 만들고 시작
    frontend 이미지 skilleatlab.azurecr.io/frontend:v4-kb5
    backend 이미지 skilleatlab.azurecr.io/backend:v4-kb5
    컨테이너 이름 backend 서비스는 반드시 backend-service (container_name 사용)
    네트워크 frontend, backend 동일 네트워크
    포트 frontend 8080:80
    실행 순서 depends_on으로 backend 먼저 실행
    데이터 유지 볼륨으로 /app/data.json 마운트

    힌트: container_name 사용법

    services:
  backend:
    container_name: backend-service
    image: ...

    검증

    1. http://192.168.56.10:8080 접속 후 글 작성
    2. docker compose exec로 backend 컨테이너에 접속해 data.json 확인
    3. docker compose down && docker compose up -d 후 글이 그대로 남아있는지 확인
