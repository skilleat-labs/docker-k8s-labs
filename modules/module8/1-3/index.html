    1-3. 레지스트리와 이미지 실습¶

    실습 목표¶

    • 공개 레지스트리(Docker Hub)와 프라이빗 레지스트리(ACR)의 차이를 이해하기
    • ACR에서 직접 이미지를 pull하며 주소 형식 체험하기
    • 이미지 캐시(레이어 재사용)가 무엇인지 실습으로 이해하기

    예상 소요 시간 (약 35분)¶

    • 0~5분: 레지스트리 개념 이해
    • 5~15분: ACR 로그인 및 이미지 pull
    • 15~25분: 이미지로 컨테이너 실행
    • 25~33분: 캐시/레이어 재사용 체감 실습
    • 33~35분: 정리

    사전 조건¶

    • Docker 설치 완료
    • Azure CLI 설치 완료
    • 강사로부터 팀 계정 정보 수령

    Azure CLI 설치 (미설치 시)¶

    Ubuntu/Debian:

    curl -sL https://aka.ms/InstallAzureCLIDeb | sudo bash

    Mac:

    brew install azure-cli

    설치 확인¶

    docker --version
az version

    1) 레지스트리란?¶

    1-1, 1-2에서는 docker run nginx:latest처럼 이미지 이름만 쓰면 자동으로 Docker Hub에서 이미지를 가져왔습니다.

    하지만 실제 기업 환경에서는 보안, 속도, 접근 제어 등의 이유로 프라이빗 레지스트리를 사용합니다.

    레지스트리 종류 비교¶

    종류 예시 특징
    공개 레지스트리 Docker Hub (docker.io) 누구나 접근 가능, 공식 이미지 제공
    프라이빗 레지스트리 Azure ACR, AWS ECR, GitHub GHCR 인증 필요, 기업 내부 이미지 관리

    주소 형식 비교¶

    # Docker Hub (레지스트리 주소 생략 가능)
nginx:latest
docker.io/library/nginx:latest   ← 전체 주소

# 프라이빗 레지스트리 (주소 명시 필수)
skilleatlab.azurecr.io/lab/nginx:latest
│                        │   │
│                        │   └── 태그
│                        └────── 이미지 경로
└─────────────────────────────── 레지스트리 주소

    핵심 포인트: 프라이빗 레지스트리에서 이미지를 가져오려면 레지스트리 주소를 앞에 붙여야 합니다.

    2) ACR 로그인 및 이미지 pull¶

    2-1. 팀 계정으로 Azure 로그인¶

    각 팀별 계정 정보:

    팀 이메일 비밀번호
    Team 01 test01@nrkim0615outlook.onmicrosoft.com 강사 제공
    Team 02 test02@nrkim0615outlook.onmicrosoft.com 강사 제공
    Team 03 test03@nrkim0615outlook.onmicrosoft.com 강사 제공 
    az login --use-device-code

    예상 출력:

    To sign in, use a web browser to open the page https://microsoft.com/devicelogin
and enter the code XXXXXXXX to authenticate.
    1. 출력된 URL(https://microsoft.com/devicelogin)을 브라우저에서 열기
    2. 출력된 코드 입력
    3. 팀 계정으로 로그인

    로그인 확인:

    az account show

    2-2. ACR 로그인¶

    az acr login --name skilleatlab

    예상 출력:

    Login Succeeded

    2-3. ACR에서 이미지 pull¶

    Docker Hub와 주소 형식이 다른 것에 주목하세요:

    docker pull skilleatlab.azurecr.io/lab/nginx:latest
docker pull skilleatlab.azurecr.io/lab/httpd:latest
docker pull skilleatlab.azurecr.io/lab/redis:latest
docker pull skilleatlab.azurecr.io/lab/alpine:latest

    목록 확인:

    docker images

    관찰 포인트:

    • REPOSITORY 컬럼에 skilleatlab.azurecr.io/...로 시작하는 이미지가 보임
    • Docker Hub 이미지(nginx:latest)와 ACR 이미지(skilleatlab.azurecr.io/lab/nginx:latest)가 별개로 관리됨
    • 내용은 같아도 주소(출처)가 다르면 다른 이미지로 취급

    3) Docker Hub에서 이미지 찾기¶

    Docker Hub: https://hub.docker.com/

    터미널에서도 검색할 수 있습니다:

    docker search nginx
docker search redis

    포인트:

    • OFFICIAL 표시가 있으면 공식 이미지
    • TAG를 명시하면 버전 고정 가능 (예: nginx:1.27)

    4) 이미지로 컨테이너 실행 실습¶

    ACR에서 받은 이미지로 컨테이너를 실행합니다.

    4-1. Nginx 실행¶

    docker run -d --name nginx-web -p 8081:80 skilleatlab.azurecr.io/lab/nginx:latest
curl -I http://localhost:8081

    4-2. Apache(httpd) 실행¶

    docker run -d --name httpd-web -p 8082:80 skilleatlab.azurecr.io/lab/httpd:latest
curl -I http://localhost:8082

    4-3. Redis 실행¶

    docker run -d --name redis-db -p 6379:6379 skilleatlab.azurecr.io/lab/redis:latest
docker logs redis-db | head

    4-4. Alpine 실행 (짧은 테스트용)¶

    Alpine이란? Alpine Linux는 약 5MB짜리 초경량 리눅스 OS 이미지입니다. nginx처럼 특정 서비스가 내장된 것도 아니고, bash만 달랑 있는 것도 아닙니다. 기본 shell(sh)과 핵심 유틸리티만 담긴 최소한의 리눅스 환경으로, 간단한 명령 실행 테스트나 다른 이미지의 베이스로 자주 사용됩니다.

    docker run --name alpine-test skilleatlab.azurecr.io/lab/alpine:latest echo "hello from alpine"

    상태 확인:

    docker ps
docker ps -a

    5) 컨테이너는 "프로세스"라는 점 이해하기¶

    핵심: 컨테이너는 내부의 메인 프로세스(PID 1)가 끝나면 같이 종료됩니다.

    5-1. 바로 종료되는 컨테이너 보기¶

    docker run --name exit-now skilleatlab.azurecr.io/lab/alpine:latest echo "bye"
docker ps -a | grep exit-now

    관찰 포인트:

    • echo 명령은 바로 끝남
    • 메인 프로세스가 끝났기 때문에 컨테이너 상태가 Exited로 바뀜

    5-2. 살아있게 유지하기 (sleep)¶

    docker run -d --name keep-alive skilleatlab.azurecr.io/lab/alpine:latest sleep 600
docker ps | grep keep-alive

    관찰 포인트:

    • sleep 600 프로세스가 600초 동안 실행됨
    • 그동안 컨테이너도 Up 상태로 유지됨

    5-3. 종료/재시작으로 동작 확인¶

    docker stop keep-alive
docker ps -a | grep keep-alive
docker start keep-alive
docker ps | grep keep-alive

    5-4. 무한 유지 패턴도 확인¶

    docker run -d --name keep-forever skilleatlab.azurecr.io/lab/alpine:latest sh -c "while true; do sleep 60; done"
docker ps | grep keep-forever

    정리:

    • 컨테이너는 VM처럼 "항상 켜져 있는 OS"가 아님
    • 메인 프로세스를 무엇으로 주느냐에 따라 바로 종료되거나 계속 실행됨

    6) 캐시(레이어 재사용) 체감 실습¶

    참고: Docker 버전에 따라 출력이 다릅니다

    Docker Engine v29(2025년 11월) 이후 신규 설치, Docker Desktop 4.34(2024년 8월) 이후 신규 설치는 기본 이미지 저장소가 containerd image store로 바뀌었습니다.

    이전 방식(graphdriver)에서는 이미 로컬에 있는 레이어가 Already exists로 표시됐지만, 새 방식(containerd)에서는 해당 레이어 라인이 출력에서 아예 사라집니다. 레이어 재사용은 동일하게 일어나지만, 눈에 보이는 출력이 달라진 것입니다.

    내 환경 확인: 

    docker info | grep "Storage Driver"
    - overlay2 → graphdriver (이전 방식, Already exists 보임) - overlayfs → containerd image store (새 방식, 라인 자체가 사라짐)

    6-1. 같은 이미지를 다시 pull¶

    이미 받은 이미지를 다시 pull하면 레이어를 재다운로드하지 않습니다.

    docker pull skilleatlab.azurecr.io/lab/nginx:latest

    예상 출력 (이미 로컬에 있을 때):

    latest: Pulling from lab/nginx
Digest: sha256:...
Status: Image is up to date for skilleatlab.azurecr.io/lab/nginx:latest

    관찰 포인트:

    • Image is up to date → 레이어를 전혀 다운로드하지 않음
    • 네트워크 트래픽 없이 즉시 완료됨

    6-2. 레이어 재사용을 직접 눈으로 확인하기¶

    pull 출력만으로는 재사용 여부가 잘 안 보이므로, docker image inspect로 레이어 해시를 직접 비교합니다.

    같은 Debian 베이스를 쓰는 두 Python 이미지를 받습니다:

    docker pull python:3.11-slim
docker pull python:3.12-slim

    레이어 목록 비교:

    docker image inspect python:3.11-slim | jq '.[0].RootFS.Layers'
docker image inspect python:3.12-slim | jq '.[0].RootFS.Layers'

    예상 출력:

    // python:3.11-slim
[
  "sha256:e6133723...",   ← Debian 베이스 레이어 (3.12와 동일)
  "sha256:384ef01e...",   ← Python 3.11 전용
  "sha256:2eeb7784...",   ← Python 3.11 전용
  "sha256:1f339219..."    ← Python 3.11 전용
]

// python:3.12-slim
[
  "sha256:e6133723...",   ← Debian 베이스 레이어 (3.11과 동일) ← 이 부분!
  "sha256:c24d0821...",   ← Python 3.12 전용
  "sha256:8d5257b6...",   ← Python 3.12 전용
  "sha256:e85497ba..."    ← Python 3.12 전용
]

    관찰 포인트:

    • 첫 번째 해시(e6133723...)가 두 이미지에서 완전히 동일 → Debian OS 베이스 레이어를 공유
    • 나머지 3개는 Python 버전마다 다름 → 언어 런타임이 달라지는 부분
    • 즉, 두 이미지를 받아도 공유 레이어는 디스크에 한 번만 저장됨
    • pull 출력에서는 이 공유 여부가 보이지 않지만, inspect로 실제 레이어 해시를 보면 증명됨

    6-3. 같은 이미지로 새 컨테이너 추가 실행¶

    docker run -d --name nginx-web-2 -p 8083:80 skilleatlab.azurecr.io/lab/nginx:latest
docker ps

    관찰 포인트:

    • 이미지를 다시 다운로드하지 않고 즉시 컨테이너가 생성됨
    • 같은 이미지 레이어를 두 컨테이너가 공유하기 때문

    6-4. 캐시가 깨지는 경우¶

    아래 중 하나면 모든 레이어를 새로 다운로드합니다.

    • 로컬 이미지와 레이어를 모두 삭제한 경우 (docker system prune)
    • 원격 이미지가 업데이트되어 digest(내용)가 바뀐 경우
    • 전혀 다른 계열의 이미지 pull

    7) 정리: 캐시된다는 뜻¶

    "이미지가 캐시된다"는 뜻은 아래와 같습니다.

    • 레이어 단위로 로컬 디스크에 저장됨
    • 동일 레이어는 재다운로드하지 않음
    • 같은 이미지를 여러 컨테이너가 공유 기반으로 사용 가능
    • 결과적으로 pull/실행 속도와 네트워크 사용량이 개선됨

    8) 실습 정리(삭제)¶

    docker rm -f nginx-web nginx-web-2 httpd-web redis-db alpine-test 2>/dev/null || true
docker rm -f exit-now keep-alive keep-forever 2>/dev/null || true

    이미지까지 지우고 싶다면:

    docker rmi skilleatlab.azurecr.io/lab/nginx:latest
docker rmi skilleatlab.azurecr.io/lab/httpd:latest
docker rmi skilleatlab.azurecr.io/lab/redis:latest
docker rmi skilleatlab.azurecr.io/lab/alpine:latest
docker rmi python:3.11-slim python:3.12-slim
