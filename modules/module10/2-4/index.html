    2-4. 미니 블로그 3-tier 배포 실습¶

    실습 목표¶

    • Frontend(Nginx) · Backend(Node.js) · Database(MySQL) 3계층 애플리케이션을 K8s에 배포한다.
    • ConfigMap과 Secret으로 설정을 분리한다.
    • Service를 통해 각 계층이 서로 통신하는 구조를 구현한다.

    전제 조건¶

    • k8s-lab 네임스페이스 사용 중

    아키텍처¶

    사용자 (브라우저)
    ↓ localhost:30000 (NodePort)
frontend-svc (NodePort :80)
    ↓
frontend Deployment (Nginx × 2)
    ↓ http://backend-svc:3000
backend-svc (ClusterIP :3000)
    ↓
backend Deployment (Node.js × 2)
    ↓ mysql-svc:3306
mysql-svc (ClusterIP :3306)
    ↓
mysql Deployment (MySQL × 1)

    1) Namespace 및 ConfigMap/Secret 배포¶

    ConfigMap¶

    blog-config.yaml 파일을 만듭니다.

    apiVersion: v1
kind: ConfigMap
metadata:
  name: blog-config
data:
  DB_HOST: mysql-svc
  DB_PORT: "3306"
  DB_NAME: blogdb
  BACKEND_URL: http://backend-svc:3000

    Secret¶

    apiVersion: v1
kind: Secret
metadata:
  name: blog-secret
type: Opaque
stringData:
  DB_PASSWORD: blogpassword123
  MYSQL_ROOT_PASSWORD: rootpassword123

    stringData를 사용하면 base64 인코딩 없이 평문으로 작성할 수 있습니다.

    kubectl apply -f blog-config.yaml
kubectl apply -f blog-secret.yaml

    2) MySQL 배포¶

    mysql-deploy.yaml 파일을 만듭니다.

    apiVersion: apps/v1
kind: Deployment
metadata:
  name: mysql
spec:
  replicas: 1
  selector:
    matchLabels:
      app: mysql
  template:
    metadata:
      labels:
        app: mysql
    spec:
      containers:
        - name: mysql
          image: mysql:8.0
          ports:
            - containerPort: 3306
          env:
            - name: MYSQL_ROOT_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: blog-secret
                  key: MYSQL_ROOT_PASSWORD
            - name: MYSQL_DATABASE
              valueFrom:
                configMapKeyRef:
                  name: blog-config
                  key: DB_NAME
            - name: MYSQL_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: blog-secret
                  key: DB_PASSWORD
          resources:
            requests:
              cpu: 100m
              memory: 256Mi
            limits:
              cpu: 500m
              memory: 512Mi
---
apiVersion: v1
kind: Service
metadata:
  name: mysql-svc
spec:
  type: ClusterIP
  selector:
    app: mysql
  ports:
    - port: 3306
      targetPort: 3306

    kubectl apply -f mysql-deploy.yaml
kubectl get pods -l app=mysql
kubectl get svc mysql-svc

    MySQL Pod가 Running 상태가 될 때까지 기다립니다.

    kubectl wait --for=condition=ready pod -l app=mysql --timeout=120s

    3) Backend(Node.js) 배포¶

    backend-deploy.yaml 파일을 만듭니다.

    apiVersion: apps/v1
kind: Deployment
metadata:
  name: backend
spec:
  replicas: 2
  selector:
    matchLabels:
      app: backend
  template:
    metadata:
      labels:
        app: backend
    spec:
      containers:
        - name: backend
          image: skilleat/order-api:v1.0.0
          ports:
            - containerPort: 8080
          envFrom:
            - configMapRef:
                name: blog-config
            - secretRef:
                name: blog-secret
          resources:
            requests:
              cpu: 100m
              memory: 128Mi
            limits:
              cpu: 300m
              memory: 256Mi
---
apiVersion: v1
kind: Service
metadata:
  name: backend-svc
spec:
  type: ClusterIP
  selector:
    app: backend
  ports:
    - port: 3000
      targetPort: 8080

    kubectl apply -f backend-deploy.yaml
kubectl get pods -l app=backend
kubectl get svc backend-svc

    4) Frontend(Nginx) 배포¶

    frontend-deploy.yaml 파일을 만듭니다.

    apiVersion: apps/v1
kind: Deployment
metadata:
  name: frontend
spec:
  replicas: 2
  selector:
    matchLabels:
      app: frontend
  template:
    metadata:
      labels:
        app: frontend
    spec:
      containers:
        - name: frontend
          image: nginx:1.25
          ports:
            - containerPort: 80
          resources:
            requests:
              cpu: 50m
              memory: 64Mi
            limits:
              cpu: 200m
              memory: 128Mi
---
apiVersion: v1
kind: Service
metadata:
  name: frontend-svc
spec:
  type: NodePort
  selector:
    app: frontend
  ports:
    - port: 80
      targetPort: 80
      nodePort: 30000

    kubectl apply -f frontend-deploy.yaml
kubectl get pods -l app=frontend
kubectl get svc frontend-svc

    5) 전체 배포 상태 확인¶

    kubectl get all

    출력 예시:

    NAME                            READY   STATUS    RESTARTS   AGE
pod/backend-xxx                 1/1     Running   0          2m
pod/frontend-xxx                1/1     Running   0          1m
pod/mysql-xxx                   1/1     Running   0          3m

NAME               TYPE        CLUSTER-IP      PORT(S)
frontend-svc       NodePort    10.96.1.10      80:30000/TCP
backend-svc        ClusterIP   10.96.1.11      3000/TCP
mysql-svc          ClusterIP   10.96.1.12      3306/TCP

NAME                  READY   UP-TO-DATE   AVAILABLE
deployment/backend    2/2     2            2
deployment/frontend   2/2     2            2
deployment/mysql      1/1     1            1

    6) 접속 확인¶

    curl http://localhost:30000

    또는 브라우저에서 http://localhost:30000 접속합니다.

    서비스 간 통신 확인¶

    Backend Pod에서 MySQL에 접근되는지 확인합니다.

    kubectl exec -it deployment/backend -- sh
# Pod 내부
curl http://backend-svc:3000/health   # 헬스체크 (이미지에 따라 경로 다름)
exit

    7) 롤링 업데이트 실습¶

    Backend를 새 버전으로 업데이트합니다.

    kubectl set image deployment/backend backend=skilleat/order-api:v2.0.0
kubectl rollout status deployment/backend
kubectl rollout history deployment/backend

    업데이트 중 Frontend는 계속 서비스됩니다.

    # 별도 터미널에서 반복 접속 (서비스 중단 없음 확인)
for i in $(seq 1 20); do curl -s -o /dev/null -w "%{http_code}\n" http://localhost:30000; done

    # 별도 터미널에서 반복 접속 (서비스 중단 없음 확인)
1..20 | ForEach-Object { curl.exe -s -o NUL -w "%{http_code}`n" http://localhost:30000 }

    정리 (리소스 삭제)¶

    kubectl delete -f frontend-deploy.yaml
kubectl delete -f backend-deploy.yaml
kubectl delete -f mysql-deploy.yaml
kubectl delete -f blog-config.yaml
kubectl delete -f blog-secret.yaml

    핵심 정리¶

    • 각 계층은 Service 이름(DNS)으로 서로를 찾습니다. Pod IP를 직접 사용하지 않습니다.
    • ConfigMap·Secret은 코드 변경 없이 환경에 따라 설정을 바꿀 수 있게 합니다.
    • Deployment replicas=2로 각 계층이 고가용성을 가집니다.

    트러블슈팅¶

    증상 확인 사항
    MySQL Pod CrashLoopBackOff Secret의 MYSQL_ROOT_PASSWORD 설정 확인
    Backend에서 DB 연결 실패 mysql-svc Service가 존재하는지, DB_HOST 값 확인
    localhost:30000 접속 불가 NodePort 30000이 사용 중인지, frontend Pod가 Running인지 확인
