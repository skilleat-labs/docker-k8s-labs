    1-7-3. 모놀리식 아키텍처 체험¶

    실습 목표¶

    • 모놀리식 아키텍처가 실제로 어떻게 동작하는지 직접 체험하기
    • 장애가 전체 서비스 중단으로 이어지는 현상 확인하기
    • 배포할 때마다 전체 서비스가 재시작되는 문제 체험하기
    • 스케일 아웃 시 데이터 불일치 문제 확인하기
    • 마이크로서비스 구조와 무엇이 다른지 비교하기

    예상 소요 시간 (약 40분)¶

    • 0~10분: 앱 실행 및 UI 탐색
    • 10~20분: 장애 전파 체험
    • 20~30분: 배포 중단 체험
    • 30~40분: 스케일링 한계 + 비교

    실습 전에 — 이 실습에 대해¶

    단순화된 모놀리식입니다

    실제 모놀리식 애플리케이션은 대부분 3-tier 구조를 사용합니다. nginx(프론트) + 하나의 큰 백엔드 앱 + MySQL(DB) 형태이고, DB는 별도 서버로 분리되어 있는 게 일반적입니다.

    이 실습의 monolith:v1은 차이를 극명하게 체험시키기 위한 단순화 버전입니다. SQLite까지 컨테이너 안에 내장해서 문제점을 더 직접적으로 느낄 수 있도록 만들었습니다.

    핵심은 인프라 구조(컨테이너 개수)가 아니라 "상품·주문·결제 등 모든 기능이 하나의 배포 단위에 묶여 있다" 는 점입니다. 어느 기능 하나를 수정해도 전체를 재배포해야 한다는 것이 모놀리식의 본질적인 한계입니다.

    사용 이미지¶

    skilleatlab.azurecr.io/monolith:v1

    이 이미지 하나에 웹 서버 · 상품 모듈 · 주문 모듈 · SQLite DB 가 모두 들어 있습니다. 별도의 DB 컨테이너나 분리된 API 서버 없이, 컨테이너 하나가 전부를 담당합니다.

    ┌─────────────────────────────────────────┐
│         monolith:v1  (컨테이너 1개)       │
│                                         │
│  ┌──────────┐  ┌──────────┐             │
│  │ 상품 모듈 │  │ 주문 모듈 │             │
│  └──────────┘  └──────────┘             │
│  ┌──────────┐  ┌──────────┐             │
│  │ 웹 서버   │  │  SQLite  │             │
│  └──────────┘  └──────────┘             │
│                                         │
│  → 모두 PID 1 단일 프로세스에서 실행      │
└─────────────────────────────────────────┘

    사전 정리¶

    docker rm -f $(docker ps -aq) 2>/dev/null || true
docker ps -a

    1) 모놀리식 앱 실행¶

    1-1. 컨테이너 실행¶

    docker run -d --name mono-cafe -p 5001:5000 \
  skilleatlab.azurecr.io/monolith:v1

    docker ps

    1-2. 브라우저에서 확인¶

    http://192.168.56.10:5001

    카페 주문 앱이 열립니다.

    1-3. 아키텍처 패널 읽기¶

    화면 하단의 ⚠️ 모놀리식 아키텍처 정보 패널을 확인합니다.

    관찰 포인트:

    • Container Hostname — 지금 요청을 처리한 컨테이너 식별자
    • Process (PID) — 모든 모듈이 실행되는 단일 프로세스 번호
    • Database — SQLite (컨테이너 내장, 외부 DB 없음)
    • 실행 중인 모듈 태그 — 상품 모듈 · 주문 모듈 · 웹 서버 · SQLite DB 가 모두 이 프로세스 안에 있음

    지금 보이는 Hostname과 PID를 기억해두세요. 3)에서 재배포 후 이 값이 어떻게 바뀌는지 비교할 것입니다.

    1-4. 주문 넣어보기¶

    메뉴 카드에서 아무 상품이나 3~4개 주문합니다. 오른쪽 사이드바에 주문 내역이 쌓이는 것을 확인합니다.

    2) 장애 = 전체 서비스 중단¶

    모놀리식의 핵심 문제입니다. 한 기능에 문제가 생기면 전체가 멈춥니다.

    2-1. 컨테이너 일시 정지¶

    docker pause mono-cafe

    브라우저를 새로고침합니다.

    http://192.168.56.10:5001

    관찰 포인트:

    • 상품 목록 → 응답 없음
    • 주문 내역 → 응답 없음
    • 상품 기능이 문제가 아니라 주문 기능이 문제여도, 상품 기능도 함께 멈춤

    2-2. 복구¶

    docker unpause mono-cafe

    새로고침하면 다시 정상 동작합니다.

    마이크로서비스였다면?

    주문 서비스 컨테이너만 문제가 생겼다면, 상품 목록은 계속 보입니다. 장애가 해당 서비스에만 격리됩니다.

    3) 배포 = 전체 서비스 재시작 + 데이터 유실¶

    기능 하나를 수정해도 앱 전체를 교체해야 합니다.

    docker restart vs 실제 배포

    docker restart는 같은 컨테이너를 재시작하는 것이라 파일시스템이 유지됩니다. 실제 배포는 컨테이너를 삭제하고 새 버전으로 교체하는 과정입니다. docker rm + docker run 으로 시뮬레이션합니다.

    3-1. 주문 몇 개 넣어두기¶

    브라우저에서 주문 내역을 2~3건 만들어둡니다.

    http://192.168.56.10:5001

    3-2. 브라우저를 열어두고 재배포 시뮬레이션¶

    터미널에서 아래 명령을 빠르게 실행합니다.

    docker rm -f mono-cafe && \
  docker run -d --name mono-cafe -p 5001:5000 \
  skilleatlab.azurecr.io/monolith:v1

    명령 실행 직후 브라우저를 새로고침합니다.

    관찰 포인트:

    • 컨테이너가 삭제되는 순간부터 새 컨테이너가 뜰 때까지 요청이 실패합니다 (연결 거부)
    • 재배포 완료 후 주문 내역이 모두 사라집니다 (SQLite가 컨테이너 내부에 있어 컨테이너 교체 시 함께 삭제됨)

    두 가지 문제가 동시에

    배포 시 서비스 중단 + 데이터 유실 이 함께 발생합니다. 실제 운영 환경에서는 이 둘 모두 치명적입니다.

    3-3. 아키텍처 패널에서 hostname 확인¶

    http://192.168.56.10:5001

    재배포 전과 비교해 hostname이 바뀐 것을 확인합니다. 완전히 새로운 컨테이너가 올라왔다는 의미입니다.

    4) 스케일링 한계¶

    트래픽이 늘었다고 가정합니다. 두 번째 인스턴스를 띄웁니다.

    4-1. 두 번째 인스턴스 실행¶

    docker run -d --name mono-cafe-2 -p 5002:5000 \
  skilleatlab.azurecr.io/monolith:v1

    4-2. 각각 주문 넣기¶

    첫 번째 인스턴스에서 주문합니다.

    http://192.168.56.10:5001

    아메리카노, 카페라떼 주문 → 주문 내역 2건 확인

    두 번째 인스턴스에서 확인합니다.

    http://192.168.56.10:5002

    관찰 포인트:

    • 5001에서 넣은 주문이 5002에서는 보이지 않습니다
    • 각 컨테이너가 자신의 SQLite를 따로 가지고 있기 때문입니다
    • 아키텍처 패널에서 Hostname이 다른 것도 확인합니다

    5002에서도 별도로 주문해보면, 두 인스턴스가 완전히 다른 데이터를 가집니다.

    모놀리식을 단순 복제하면

    각 컨테이너가 독립된 DB를 가지게 됩니다. 사용자가 어느 서버로 접속되느냐에 따라 보이는 데이터가 달라집니다. 이것이 상태(state)를 내장한 모놀리식을 수평 확장할 때의 근본적인 문제입니다.

    4-3. 정리¶

    docker rm -f mono-cafe mono-cafe-2

    5) 모놀리식 vs 마이크로서비스 비교¶

    오늘 체험한 내용을 정리하면 다음과 같습니다.

    상황 모놀리식 마이크로서비스
    한 기능에 장애 발생 전체 서비스 중단 해당 서비스만 영향
    배포 시 전체 재시작, 중단 시간 발생 해당 서비스만 재시작
    스케일 아웃 전체를 복제해야 함 필요한 서비스만 늘릴 수 있음
    데이터 공유 복제 시 각자 다른 DB 공유 DB / 서비스간 API로 해결
    개발 작을 때는 단순함 서비스가 커질수록 유리

    그래서 K8s가 필요합니다

    마이크로서비스 구조에서도 문제는 있습니다. 서비스가 많아질수록 어떤 컨테이너가 어디서 실행되는지, 장애 시 자동으로 복구되는지, 트래픽에 따라 자동으로 스케일되는지 를 수동으로 관리하기 어렵습니다. 이것이 Kubernetes가 등장한 이유입니다.

    📝 핵심 요약¶

    1. 모놀리식 — 모든 기능이 하나의 프로세스에서 실행, 단순하지만 규모가 커지면 한계
    2. 장애 격리 불가 — 한 기능의 문제가 전체 서비스 중단으로 이어짐
    3. 배포 비용 — 작은 변경도 전체 재시작 필요, 서비스 중단 시간 발생
    4. 스케일링 비효율 — 특정 기능만 늘릴 수 없고, 복제 시 데이터 동기화 문제
    5. 마이크로서비스로 가는 이유 — 위 문제들을 서비스 단위 격리로 해결
