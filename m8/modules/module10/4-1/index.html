    4-1. Probe 설정 실습 (Liveness · Readiness · Startup)¶

    실습 목표¶

    • Liveness Probe로 교착 상태 컨테이너를 자동 재시작하는 것을 확인한다.
    • Readiness Probe로 준비되지 않은 Pod를 Service에서 제외하는 것을 확인한다.
    • Startup Probe로 느리게 시작하는 앱의 초기화를 보호한다.

    전제 조건¶

    • k8s-lab 네임스페이스 사용 중

    1) 기본 개념 정리¶

    Probe 실패 시 동작 주요 용도
    Liveness 컨테이너 재시작 교착 상태·무한 루프 감지
    Readiness Service 엔드포인트에서 제거 (재시작 없음) DB 연결 전 트래픽 차단
    Startup 성공 전까지 Liveness/Readiness 비활성화 느린 초기화 앱 보호

    2) Liveness Probe 실습¶

    정상 동작하는 Liveness Probe¶

    pod-liveness-ok.yaml 파일을 만듭니다.

    apiVersion: v1
kind: Pod
metadata:
  name: liveness-ok
spec:
  containers:
    - name: app
      image: busybox:1.36
      command:
        - sh
        - -c
        - "touch /tmp/healthy && sleep 3600"
      livenessProbe:
        exec:
          command:
            - cat
            - /tmp/healthy
        initialDelaySeconds: 5
        periodSeconds: 5
        failureThreshold: 3

    kubectl apply -f pod-liveness-ok.yaml
kubectl get pod liveness-ok -w

    Pod가 Running 상태를 유지합니다. Ctrl+C 종료.

    Liveness Probe 실패 → 재시작¶

    pod-liveness-fail.yaml 파일을 만듭니다.

    apiVersion: v1
kind: Pod
metadata:
  name: liveness-fail
spec:
  containers:
    - name: app
      image: busybox:1.36
      command:
        - sh
        - -c
        - "touch /tmp/healthy && sleep 30 && rm /tmp/healthy && sleep 3600"
      livenessProbe:
        exec:
          command:
            - cat
            - /tmp/healthy
        initialDelaySeconds: 5
        periodSeconds: 5
        failureThreshold: 3

    30초 후 /tmp/healthy 파일이 삭제되어 Probe가 실패합니다.

    kubectl apply -f pod-liveness-fail.yaml
kubectl get pod liveness-fail -w

    30초~1분 후 RESTARTS 숫자가 증가하는 것을 확인합니다.

    kubectl describe pod liveness-fail | grep -A 5 Events

    kubectl describe pod liveness-fail | Select-String -Pattern "Events" -A 5

    이벤트에서 Liveness probe failed → Killing → Started 순서로 확인합니다.

    3) Readiness Probe 실습¶

    deploy-readiness.yaml 파일을 만듭니다.

    apiVersion: apps/v1
kind: Deployment
metadata:
  name: readiness-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: readiness-app
  template:
    metadata:
      labels:
        app: readiness-app
    spec:
      containers:
        - name: app
          image: nginx:1.25
          ports:
            - containerPort: 80
          readinessProbe:
            httpGet:
              path: /
              port: 80
            initialDelaySeconds: 5
            periodSeconds: 5
            failureThreshold: 3
---
apiVersion: v1
kind: Service
metadata:
  name: readiness-svc
spec:
  selector:
    app: readiness-app
  ports:
    - port: 80
      targetPort: 80

    kubectl apply -f deploy-readiness.yaml
kubectl get pods -l app=readiness-app
kubectl get endpoints readiness-svc

    3개 Pod가 모두 READY 1/1이 되면 Endpoints에 3개의 IP가 표시됩니다.

    Pod 하나의 Readiness를 강제로 실패시키기¶

    # Pod 이름 확인
kubectl get pods -l app=readiness-app

# Pod 내부에서 Nginx 중지 (Readiness Probe 실패 유발)
kubectl exec <pod이름> -- nginx -s stop

    kubectl get pods -l app=readiness-app    # READY 0/1 으로 변경
kubectl get endpoints readiness-svc     # 해당 Pod IP가 Endpoints에서 제거됨

    포인트: Pod는 Running 상태이지만 READY=0/1로 Service Endpoint에서 제외되어 트래픽을 받지 않습니다.

    4) Startup Probe 실습¶

    느리게 시작하는 앱(예: JVM 기동 30초 필요)을 시뮬레이션합니다.

    pod-startup.yaml 파일을 만듭니다.

    apiVersion: v1
kind: Pod
metadata:
  name: startup-app
spec:
  containers:
    - name: app
      image: busybox:1.36
      command:
        - sh
        - -c
        - "sleep 20 && touch /tmp/started && sleep 3600"
      startupProbe:
        exec:
          command:
            - cat
            - /tmp/started
        failureThreshold: 30    # 30회 × periodSeconds(10s) = 최대 300초 대기
        periodSeconds: 10
      livenessProbe:
        exec:
          command:
            - cat
            - /tmp/started
        initialDelaySeconds: 0
        periodSeconds: 10
        failureThreshold: 3

    kubectl apply -f pod-startup.yaml
kubectl get pod startup-app -w

    처음 20초간 Startup Probe가 실패하지만 Pod는 재시작되지 않습니다 (Liveness는 비활성화). 20초 후 /tmp/started 파일이 생성되고 Startup Probe가 성공하면 READY 1/1이 됩니다.

    5) 모든 Probe 조합¶

    pod-all-probes.yaml 파일을 만듭니다.

    apiVersion: v1
kind: Pod
metadata:
  name: all-probes
spec:
  containers:
    - name: app
      image: nginx:1.25
      ports:
        - containerPort: 80
      startupProbe:
        httpGet:
          path: /
          port: 80
        failureThreshold: 30
        periodSeconds: 10
      readinessProbe:
        httpGet:
          path: /
          port: 80
        initialDelaySeconds: 0
        periodSeconds: 5
        failureThreshold: 3
      livenessProbe:
        httpGet:
          path: /
          port: 80
        initialDelaySeconds: 0
        periodSeconds: 10
        failureThreshold: 3

    kubectl apply -f pod-all-probes.yaml
kubectl describe pod all-probes | grep -A 15 "Liveness\|Readiness\|Startup"

    정리 (리소스 삭제)¶

    kubectl delete pod liveness-ok liveness-fail startup-app all-probes
kubectl delete -f deploy-readiness.yaml

    실무 설정 가이드¶

    Probe 권장 설정
    Startup failureThreshold × periodSeconds ≥ 최대 초기화 시간
    Readiness initialDelaySeconds 작게, periodSeconds 5~10s
    Liveness initialDelaySeconds 충분히 (DB 연결 시간 고려), failureThreshold 3~5

    트러블슈팅¶

    증상 확인 사항
    Pod가 계속 재시작 Liveness Probe 실패 원인 확인 (kubectl describe pod)
    Pod는 Running이지만 READY=0/1 Readiness Probe 실패 — 앱이 아직 준비 안된 것
    Pod가 initialDelaySeconds 전에 재시작 Startup Probe 추가 또는 initialDelaySeconds 늘리기
