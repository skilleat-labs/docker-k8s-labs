    4-3. HPA (Horizontal Pod Autoscaler) 실습¶

    실습 목표¶

    • Metrics Server를 설치하여 HPA의 전제 조건을 갖춘다.
    • CPU 사용률 기반 HPA를 설정하고 부하 발생 시 자동 확장을 확인한다.
    • 부하 제거 후 Pod가 자동으로 축소되는 것을 확인한다.

    전제 조건¶

    • k8s-lab 네임스페이스 사용 중
    • Deployment에 resources.requests 설정 필수 (HPA 동작 조건)

    1) Metrics Server 설치¶

    HPA는 Metrics Server에서 CPU/Memory 수치를 받아 동작합니다.

    kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml

    Docker Desktop / Rancher Desktop 설정 조정¶

    로컬 환경에서는 TLS 인증서 검증을 비활성화해야 합니다.

    kubectl patch deployment metrics-server -n kube-system \
  --type='json' \
  -p='[{"op": "add", "path": "/spec/template/spec/containers/0/args/-", "value": "--kubelet-insecure-tls"}]'

    설치 확인 (1~2분 소요):

    kubectl get pods -n kube-system -l k8s-app=metrics-server
kubectl top nodes
kubectl top pods -A

    kubectl top 명령어가 정상 출력되면 Metrics Server 준비 완료입니다.

    2) HPA 대상 Deployment 배포¶

    deploy-hpa-target.yaml 파일을 만듭니다.

    apiVersion: apps/v1
kind: Deployment
metadata:
  name: hpa-app
spec:
  replicas: 1
  selector:
    matchLabels:
      app: hpa-app
  template:
    metadata:
      labels:
        app: hpa-app
    spec:
      containers:
        - name: app
          image: k8s.gcr.io/hpa-example
          ports:
            - containerPort: 80
          resources:
            requests:
              cpu: 100m       # HPA 동작에 필수
              memory: 64Mi
            limits:
              cpu: 500m
              memory: 128Mi
---
apiVersion: v1
kind: Service
metadata:
  name: hpa-svc
spec:
  selector:
    app: hpa-app
  ports:
    - port: 80
      targetPort: 80

    kubectl apply -f deploy-hpa-target.yaml
kubectl get pods -l app=hpa-app

    k8s.gcr.io/hpa-example은 / 경로에 HTTP 요청이 올 때 CPU를 의도적으로 사용하는 테스트 이미지입니다.

    3) HPA 생성¶

    방법 A — kubectl 명령어¶

    kubectl autoscale deployment hpa-app \
  --cpu-percent=50 \
  --min=1 \
  --max=10

    방법 B — YAML 파일¶

    hpa.yaml 파일을 만듭니다.

    apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: hpa-app
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: hpa-app
  minReplicas: 1
  maxReplicas: 10
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 50
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 60    # 1분 안정화 후 축소

    kubectl apply -f hpa.yaml
kubectl get hpa

    출력 예시:

    NAME      REFERENCE            TARGETS         MINPODS   MAXPODS   REPLICAS   AGE
hpa-app   Deployment/hpa-app   <unknown>/50%   1         10        1          30s

    TARGETS의 왼쪽이 <unknown>에서 실제 수치로 바뀔 때까지 1~2분 기다립니다.

    kubectl get hpa -w

    4) 부하 발생 — 자동 확장 확인¶

    터미널 1 — HPA 상태 감시¶

    kubectl get hpa -w

    터미널 2 — Pod 수 감시¶

    kubectl get pods -l app=hpa-app -w

    터미널 3 — 부하 발생¶

    busybox Pod를 실행하여 hpa-svc에 반복 HTTP 요청을 보냅니다.

    kubectl run load-generator \
  --image=busybox:1.36 \
  --restart=Never \
  -- sh -c "while true; do wget -q -O- http://hpa-svc; done"

    결과 확인¶

    터미널 1에서 CPU 사용률이 상승하고 REPLICAS가 증가하는 것을 확인합니다.

    NAME      REFERENCE            TARGETS    MINPODS   MAXPODS   REPLICAS
hpa-app   Deployment/hpa-app   12%/50%    1         10        1
hpa-app   Deployment/hpa-app   85%/50%    1         10        1
hpa-app   Deployment/hpa-app   85%/50%    1         10        3
hpa-app   Deployment/hpa-app   72%/50%    1         10        5

    HPA는 약 15초마다 메트릭을 확인하고 replica 수를 조정합니다.

    5) 부하 제거 — 자동 축소 확인¶

    부하 발생 Pod를 중지합니다.

    kubectl delete pod load-generator

    터미널 1·2에서 CPU 사용률이 감소하고 stabilizationWindowSeconds(60초) 후 Replica가 1로 축소되는 것을 확인합니다.

    6) 관찰 기록표¶

    아래 표를 채워가면서 실습합니다.

    시점 CPU 사용률 Replica 수 비고
    부하 발생 전 1 초기 상태
    부하 발생 후 1분
    부하 발생 후 3분
    부하 제거 직후
    부하 제거 후 2분 축소 시작
    최종 안정 1 Scale-in 완료

    7) HPA 상태 상세 확인¶

    kubectl describe hpa hpa-app

    확인 항목:

    • Current Metrics: 현재 CPU 사용률
    • Min replicas / Max replicas
    • Events: 스케일 업/다운 이벤트 기록

    정리 (리소스 삭제)¶

    kubectl delete hpa hpa-app
kubectl delete -f deploy-hpa-target.yaml
kubectl delete pod load-generator --ignore-not-found

    HPA 동작 공식¶

    목표 Replica = ceil(현재 Replica × (현재 메트릭 / 목표 메트릭))

예시:
  현재 Replica: 2
  현재 CPU: 90%
  목표 CPU: 50%
  → ceil(2 × 90/50) = ceil(3.6) = 4

    HPA 전제 조건 체크리스트¶

    • [ ] Deployment에 resources.requests.cpu 설정
    • [ ] Metrics Server 설치 및 정상 동작 (kubectl top nodes)
    • [ ] HPA의 scaleTargetRef가 실제 Deployment 이름과 일치

    트러블슈팅¶

    증상 확인 사항
    TARGETS가 계속 <unknown> Metrics Server Pod 상태 확인, --kubelet-insecure-tls 패치 적용 확인
    CPU 100%인데 Scale-up 안됨 Deployment에 resources.requests.cpu 설정 여부 확인
    Scale-down이 느림 stabilizationWindowSeconds 값 확인 (기본 300초)
    kubectl top 오류 Metrics Server Pod가 Running인지 확인
