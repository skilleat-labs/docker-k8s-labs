    2-1. Pod & Deployment 실습¶

    실습 목표¶

    • kubectl run과 YAML로 Pod를 생성하고 차이를 이해한다.
    • 멀티 컨테이너 Pod 구조를 실습한다.
    • ReplicaSet이 Pod를 자동으로 복구하는 동작을 확인한다.
    • Label의 역할과 --show-labels 옵션을 익힌다.
    • Deployment로 롤링 업데이트와 롤백을 실습한다.

    공식 문서 참고¶

    실습 중 모르는 필드나 옵션이 생기면 공식 문서를 참고합니다.

    • Pod 공식 문서
    • Deployment 공식 문서
    • kubectl 치트시트

    1) kubectl run으로 Pod 빠르게 생성¶

    YAML 없이 명령어 한 줄로 Pod를 만들 수 있습니다.

    kubectl run nginx-pod --image=nginx:1.25

    상태 확인:

    kubectl get pods
kubectl get pods -o wide    # Pod IP 포함

    dry-run으로 YAML 미리 보기¶

    --dry-run=client -o yaml 옵션을 붙이면 실제로 생성하지 않고 YAML만 출력합니다. YAML 작성이 익숙하지 않을 때 시작점으로 활용할 수 있습니다.

    kubectl run test --image=nginx --dry-run=client -o yaml

    출력 예시:

    apiVersion: v1
kind: Pod
metadata:
  creationTimestamp: null
  labels:
    run: test
  name: test
spec:
  containers:
  - image: nginx
    name: test
    resources: {}
  dnsPolicy: ClusterFirst
  restartPolicy: Always

    이 출력을 파일로 저장해서 편집하면 됩니다.

    kubectl run test --image=nginx --dry-run=client -o yaml > pod-test.yaml

    2) YAML로 Pod 생성¶

    pod-nginx.yaml 파일을 만듭니다.

    apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
  labels:
    app: nginx
    env: lab
spec:
  containers:
    - name: nginx
      image: nginx:1.25
      ports:
        - containerPort: 80

    kubectl apply -f pod-nginx.yaml
kubectl get pods
kubectl describe pod nginx-pod

    describe 출력에서 확인:

    • Status: Running
    • IP: Pod에 할당된 내부 IP
    • Labels: 부여된 label 목록
    • Events: 이미지 pull → 컨테이너 시작 과정

    로그 확인¶

    kubectl logs nginx-pod
kubectl logs nginx-pod -f    # 실시간 스트리밍

    Pod 내부 접속¶

    kubectl exec -it nginx-pod -- bash
# 컨테이너 내부
nginx -v
curl localhost
exit

    curl 이미지로 Pod 간 통신 확인¶

    nginx-pod의 IP를 확인한 뒤, curl 이미지를 실행해 해당 IP로 요청을 보냅니다.

    kubectl get pods -o wide    # nginx-pod의 IP 확인 (예: 10.1.0.5)

    kubectl run curl-test --image=curlimages/curl:latest --restart=Never -it --rm \
  -- curl http://10.1.0.5

    nginx 응답 HTML이 출력되면 성공입니다. --rm 옵션으로 종료 시 Pod가 자동 삭제됩니다.

    Pod 삭제 후 IP 변화 관찰¶

    kubectl delete pod nginx-pod
kubectl apply -f pod-nginx.yaml
kubectl get pods -o wide    # 새 IP가 할당됨을 확인

    포인트: Pod를 재생성하면 IP가 바뀝니다. 그래서 Service가 필요합니다.

    3) 멀티 컨테이너 Pod¶

    하나의 Pod에 컨테이너를 여러 개 넣을 수 있습니다. 같은 네트워크와 스토리지를 공유하므로 localhost로 서로 통신합니다.

    command 필드란?¶

    command는 컨테이너가 시작될 때 실행할 명령어를 직접 지정하는 필드입니다. Docker의 ENTRYPOINT에 해당합니다.

    필드 Docker 대응 설명
    command ENTRYPOINT 실행할 명령어 (이미지 기본값 덮어씀)
    args CMD 명령어에 전달할 인자

    command를 지정하지 않으면 이미지에 정의된 기본 명령어가 실행됩니다. 아래 예시에서 sidecar 컨테이너는 이미지 기본 동작 대신 sh -c "while true; ..." 쉘 스크립트를 실행합니다.

    command: ["sh", "-c", "while true; do echo $(date) $(curl -s localhost); sleep 5; done"]
#          ↑실행파일   ↑sh에 전달되는 인자: 5초마다 localhost(=nginx)에 curl 요청

    pod-multi.yaml 파일을 만듭니다.

    apiVersion: v1
kind: Pod
metadata:
  name: multi-pod
  labels:
    app: multi
spec:
  containers:
    - name: nginx
      image: nginx:1.25
      ports:
        - containerPort: 80
    - name: sidecar
      image: curlimages/curl:latest
      command: ["sh", "-c", "while true; do echo $(date) $(curl -s localhost); sleep 5; done"]

    kubectl apply -f pod-multi.yaml
kubectl get pods    # READY 2/2 확인

    각 컨테이너 로그를 따로 확인합니다.

    kubectl logs multi-pod -c nginx      # nginx 컨테이너 로그
kubectl logs multi-pod -c sidecar    # sidecar 컨테이너 로그

    sidecar가 localhost로 nginx에 접근하여 응답을 받는 것을 확인합니다.

    특정 컨테이너 내부 접속:

    kubectl exec -it multi-pod -c nginx -- bash

    4) Label과 --show-labels¶

    Label은 K8s 리소스에 붙이는 키-값 태그입니다. Deployment가 Pod를 선택하거나 Service가 트래픽을 보낼 대상을 찾을 때 label을 기준으로 합니다.

    label 확인¶

    kubectl get pods --show-labels

    출력 예시:

    NAME        READY   STATUS    RESTARTS   AGE   LABELS
nginx-pod   1/1     Running   0          2m    app=nginx,env=lab
multi-pod   2/2     Running   0          1m    app=multi

    label 기준으로 필터링¶

    -l 옵션으로 특정 label을 가진 Pod만 조회할 수 있습니다.

    kubectl get pods -l app=nginx
kubectl get pods -l env=lab
kubectl get pods -l app=nginx,env=lab    # 두 조건 모두 만족

    label 추가 / 변경 / 삭제¶

    kubectl label pod nginx-pod version=v1          # label 추가
kubectl label pod nginx-pod env=production --overwrite   # label 변경
kubectl label pod nginx-pod version-            # label 삭제 (키 뒤에 - 붙임)
kubectl get pods --show-labels                  # 변경 결과 확인

    5) ReplicaSet 실습¶

    ReplicaSet은 지정한 수만큼 Pod를 항상 유지합니다. Deployment를 만들면 내부적으로 ReplicaSet이 생성됩니다.

    replicaset.yaml 파일을 만듭니다.

    apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: nginx-rs
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx-rs
  template:
    metadata:
      labels:
        app: nginx-rs
    spec:
      containers:
        - name: nginx
          image: nginx:1.25
          ports:
            - containerPort: 80

    kubectl apply -f replicaset.yaml
kubectl get replicasets
kubectl get pods --show-labels

    Pod 삭제 후 자동 복구 확인¶

    Pod 하나를 삭제합니다.

    kubectl get pods -l app=nginx-rs
kubectl delete pod <pod이름 중 하나>
kubectl get pods -l app=nginx-rs -w    # 즉시 새 Pod가 생성되는 것을 실시간 확인

    Ctrl+C로 watch 모드 종료.

    ReplicaSet이 replicas: 3을 유지하기 위해 삭제된 Pod를 즉시 새로 만듭니다.

    Label을 바꾸면 ReplicaSet 관리에서 빠짐¶

    ReplicaSet은 label selector로 Pod를 관리합니다. Pod의 label을 바꾸면 ReplicaSet이 더 이상 그 Pod를 자신의 Pod로 인식하지 않아 새 Pod를 추가로 생성합니다.

    # Pod 중 하나의 label 변경
kubectl label pod <pod이름> app=orphan --overwrite
kubectl get pods --show-labels    # Pod가 4개가 됨 (고아 Pod 1개 + ReplicaSet 관리 3개)

    확인 후 고아 Pod 삭제:

    kubectl delete pod -l app=orphan

    6) Deployment 생성¶

    Deployment는 ReplicaSet을 한 단계 감싸며, 롤링 업데이트·롤백 기능을 추가로 제공합니다.

    dry-run으로 Deployment YAML 뽑기¶

    Pod와 마찬가지로 --dry-run=client -o yaml로 Deployment YAML의 시작점을 만들 수 있습니다.

    kubectl create deployment rollout-deploy \
  --image=skilleat/rollout-demo:v1.0.0 \
  --replicas=3 \
  --dry-run=client -o yaml

    출력 예시:

    apiVersion: apps/v1
kind: Deployment
metadata:
  name: rollout-deploy
spec:
  replicas: 3
  selector:
    matchLabels:
      app: rollout-deploy
  template:
    metadata:
      labels:
        app: rollout-deploy
    spec:
      containers:
      - image: skilleat/rollout-demo:v1.0.0
        name: rollout-demo

    파일로 저장한 뒤 필요한 부분(label, containerPort 등)을 편집합니다.

    kubectl create deployment rollout-deploy \
  --image=skilleat/rollout-demo:v1.0.0 \
  --replicas=3 \
  --dry-run=client -o yaml > deploy-rollout.yaml

    YAML 편집 후 배포¶

    저장된 deploy-rollout.yaml을 열어 containerPort를 추가하고 label을 정리합니다. 완성된 파일은 아래와 같습니다. v1.0.0은 파란색 배경 페이지를 반환합니다.

    apiVersion: apps/v1
kind: Deployment
metadata:
  name: rollout-deploy
spec:
  replicas: 3
  selector:
    matchLabels:
      app: rollout
  template:
    metadata:
      labels:
        app: rollout
    spec:
      containers:
        - name: app
          image: skilleat/rollout-demo:v1.0.0
          ports:
            - containerPort: 80

    kubectl apply -f deploy-rollout.yaml
kubectl get deployments
kubectl get replicasets
kubectl get pods --show-labels

    Deployment → ReplicaSet → Pod 계층 구조를 확인합니다.

    kubectl describe deployment rollout-deploy    # NewReplicaSet 이름 확인
kubectl describe replicaset <rs이름>          # 관리 중인 Pod 확인

    배포 후 port-forward로 현재 버전을 브라우저에서 확인합니다. 파란색 페이지가 표시됩니다.

    kubectl port-forward deployment/rollout-deploy 8080:80

    http://localhost:8080 접속 후 Ctrl+C로 종료.

    7) 롤링 업데이트¶

    v2.0.0으로 업데이트합니다. v2.0.0은 초록색 배경 페이지를 반환합니다.

    kubectl set image deployment/rollout-deploy app=skilleat/rollout-demo:v2.0.0
kubectl rollout status deployment/rollout-deploy
kubectl get pods -w    # 구 Pod(파란색) 종료, 새 Pod(초록색) 생성 교체 과정 확인

    Ctrl+C로 watch 모드 종료.

    완료 후 다시 port-forward로 확인합니다. 초록색 페이지로 바뀌어 있어야 합니다.

    kubectl port-forward deployment/rollout-deploy 8080:80

    이미지 버전 확인:

    kubectl describe deployment rollout-deploy | grep Image

    8) 롤백¶

    초록색 → 파란색으로 되돌립니다.

    kubectl rollout history deployment/rollout-deploy    # 변경 히스토리
kubectl rollout undo deployment/rollout-deploy       # 이전 버전(v1.0.0)으로 복귀
kubectl rollout status deployment/rollout-deploy
kubectl describe deployment rollout-deploy | grep Image    # v1.0.0 복귀 확인

    port-forward로 다시 확인하면 파란색 페이지로 돌아옵니다.

    kubectl port-forward deployment/rollout-deploy 8080:80

    9) 스케일 조정¶

    kubectl scale deployment rollout-deploy --replicas=5
kubectl get pods
kubectl scale deployment rollout-deploy --replicas=2
kubectl get pods

    정리 (리소스 삭제)¶

    kubectl delete -f replicaset.yaml
kubectl delete -f deploy-rollout.yaml
kubectl delete -f pod-nginx.yaml
kubectl delete -f pod-multi.yaml

    핵심 정리¶

    항목 Pod 단독 ReplicaSet Deployment
    복제본 관리 수동 자동 자동
    장애 복구 없음 자동 재생성 자동 재생성
    롤링 업데이트 불가 불가 내장 지원
    실사용 권장 디버깅용 거의 직접 사용 안 함 프로덕션 배포
